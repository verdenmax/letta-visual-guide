第三部分 · 记忆系统Part 3 · The memory system

上下文压缩与记忆压力Context compaction & memory pressure

第 5 课摊清过一笔账：上下文窗口又小又贵，"把历史一股脑全塞进去"必然失败。当时我们停在两件可操作的事上——可度量（ContextWindowOverview 把窗口拆成一本 token 账本）、与可触发动作（逼近阈值就压缩）。那一课讲的是"度量"那一半，这一课补上"动手"那一半。

窗口快满了怎么办？答案是压缩：把较旧的对话总结成一段摘要，再把原话换出窗外。五个关键词串起整课——90% 阈值、compact_messages、滑窗摘要、role=summary 的摘要消息，以及"核心记忆压不动"的系统提示溢出。这是第三部分的收尾，也是整套记忆系统真正闭环的一环。

🔌 生活类比
把"记忆压力"想成一张快堆满的办公桌。你正在桌上摊开资料办事，桌面就那么大——纸越摞越高，眼看连放笔的地方都没了。聪明的做法不是把旧纸直接扔掉（万一还要查呢？），而是花一分钟把较旧的那叠读一遍、写成一张"会议纪要"：要点都落在纪要上，原始的厚厚一叠则收进抽屉。于是桌面腾出大半，你又能继续摊开新资料，而那张纪要还摆在眼前提醒你"前面聊到哪了"。原纸没销毁，要核对时还能从抽屉翻出来；但日常推进，靠那张压缩过的纪要就够了。Letta 的上下文压缩干的正是这件事——只不过"较旧的那叠纸"是旧对话消息，"会议纪要"是一段自动生成的摘要，而决定"什么时候该收桌子"的，是一条画在 90% 处的水位线。记住这张桌子，下面每个机制你都能对号入座。
🌍 宏观理解
一句话抓住本课：当在窗 token 逼近上限的 90%，Letta 就把较旧的对话压成一段摘要、换出原话，腾出空间继续聊——这套机制把"记忆压力"当成操作系统问题来管。触发靠一条水位线：每次模型调用后顺手记下用量（context_token_estimate），一旦超过 context_window × 0.9 这个阈值（get_compaction_trigger_threshold），就调 compact 动手。动手的方式是滑窗摘要：把较旧的一段消息交给一个 summarizer，递归地总结成一条 role=summary 的摘要消息插回上下文，原话退到窗外但仍留在 recall 里可搜。整个过程还能（打开开关时）抛出一个可见事件，让客户端看到"我刚做了一次压缩"。唯一压不动的是核心记忆与系统提示——它们是 system 的一部分，summarizer 只清理对话、绝不动 core，所以另有一条系统提示溢出的特判路径来兜底。

先把这一课接回第 5 课

第 5 课画过一条闭环：度量 → 判定 → 压缩 → 重建前缀。前两步（数清谁占了多少 token、判断是否逼近阈值）那一课已经讲透；这一课从第三步"压缩"接着往下挖，看它具体怎么动手、动完之后又发生了什么。

🧠一句话定位：第 5 课是"量得出"（ContextWindowOverview 那本账本），第 12 课是"压得动"（compact 那把剪刀）。同一条闭环的两半，这一课补齐后半段。

为什么非压不可？因为 recall（第 11 课）保证了"对话不丢"，却没解决"在窗装不下"。消息只增不减，迟早撑爆预算；压缩就是那把让窗口"转得动"的阀门——它不删历史，只把较旧的一段换成更短的摘要。

📚→🗜️→📝一段摘要
窗口快满（~90%）：把较旧的一摞消息压成一张"会议纪要"，腾出空间继续聊；原话退到窗外但仍可搜

90% 阈值：什么时候开始压

压缩不等窗口"真满"才动手——那样很容易撞上"prompt 太长"的报错。Letta 留了 10% 余量：在上限的 90% 就触发。这个阈值由 get_compaction_trigger_threshold 算出，等于 context_window × 0.9。

用量 0% · 安心聊 70% · 还宽裕 90% · 触发阈值
动作 逼近 90% → compact：把较旧消息压成摘要 换出原话 · 插入 summary
结果 用量回落 → 继续聊；原话仍在 recall 可搜
letta/services/summarizer/thresholds.py · letta/constants.py压缩触发阈值（简化）
# 触发阈值 = 上下文窗口 × 0.9（留 10% 余量，避免撞上 "prompt 太长"）
SUMMARIZATION_TRIGGER_MULTIPLIER = 0.9   # letta/constants.py

def get_compaction_trigger_threshold(llm_config, *, force_proactive=False):
    # 不区分模型，一律取 上限 × 0.9（force_proactive 目前不改变结果）
    return int(llm_config.context_window * SUMMARIZATION_TRIGGER_MULTIPLIER)
📌阈值按窗口比例算，不是写死的某个数字：窗口 8k 就在 ~7.2k 触发，128k 就在 ~115k 触发。窗口再大，到 90% 照样压——大窗口只是把"何时压"推后，并没取消它。

触发点：用量 vs 阈值，就在 agent 主循环里

阈值在哪被用上？在 letta/agents/letta_agent_v3.py 的主循环里。运行时为了快，不每步重算整本账，而是顺手记下每次模型调用返回的 token 用量，拿这个估算值跟阈值比一下。

1

记下用量

模型一返回，就把它报告的 usage.total_tokens 存进 context_token_estimate。

2

比一比阈值

估算值 > get_compaction_trigger_threshold（×0.9）？

3

可选：抛事件

开关打开时，先 yield 一个 compaction 事件告诉客户端"要压了"。

4

动手压缩

调 self.compact(messages, ...)，把较旧消息换成摘要。

关键细节：拿来比的不是"账本精算值"，而是上一次模型调用自己报告的 usage.total_tokens。这等于让模型每轮顺带量一次自己的体重——省去重算，又足够准。一旦超阈值，就进压缩分支。

💡两条触发路径，殊途同归。一条是事后水位检查（这步聊完、用量过 90% 就压，trigger="post_step_context_check"）；另一条是真撞墙兜底（模型直接报 ContextWindowExceededError，就地压完重试，trigger="context_window_exceeded"）。两条都汇入同一个 compact。

compact_messages：把旧消息压成摘要

self.compact(...) 只是个薄壳，真正干活的是 letta/services/summarizer/compact.py::compact_messages。它拿到当前在窗消息，交给 summarizer 总结，产出三样东西：一条摘要消息、压缩后的消息列表、摘要正文。

letta/services/summarizer/compact.pycompact_messages（伪代码）
async def compact_messages(..., messages, use_summary_role=True, trigger_threshold=None, ...):
    # 1) 用 summarizer 把较旧的一段消息总结掉（滑窗 / 递归）
    summary, compacted_messages = await summarize(messages)

    # 2) 再数一遍 token，确认压完是否真降到阈值以下
    context_token_estimate = await count_tokens_with_tools(messages=compacted_messages, ...)

    # 3) 若仍超阈值，且元凶是第 0 条系统消息 -> 走溢出特判
    if context_token_estimate >= trigger_threshold:
        if tokens_of(compacted_messages[0]) >= llm_config.context_window:
            raise SystemPromptTokenExceededError(...)   # 核心记忆/系统提示太大

    return CompactResult(summary_message, compacted_messages, summary, ...)

注意第 3 步：压完之后它会再数一遍 token 验收（仍超，会先再跑一次"全量总结"兜底）。要是连这都救不回，而且一查发现"元凶是第 0 条系统消息本身"——那就不是对话太长，是核心记忆/系统提示撑爆了预算。这条岔路就是后面要讲的"系统提示溢出"。

滑窗摘要：到底压哪一段、怎么压

"压缩"具体怎么选消息？由 Summarizer（summarizer.py）负责。它有几种模式，常用这种叫滑动窗口 / 部分驱逐（partial-evict）。核心思路一句话：留最近的一截，把更早的那截总结掉。

1

算一条切割线

partial_evict_summarizer_percentage（默认 0.30）是保留比例：留最近 ~30%，把更早的 ~70% 总结掉。

2

对齐到边界

从切割点往后找第一条 assistant 消息，保证摘要插回后角色顺序合法。

3

总结被驱逐段

把 [1 : 该 assistant) 这一段交给 summarizer LLM，生成一段摘要。

4

替换 index 1

摘要放回第 1 条（第 0 条永远是系统消息），原话退出窗口。

这就是 MemGPT 论文里的递归摘要：摘要被固定放在 [1]，下一次再压时，它又会连同后来的新消息一起被再总结一遍。于是历史像滚雪球一样被一层层压进越来越浓缩的一段摘要，越久远的越凝练。

为什么从切割点"往后找 assistant"？因为摘要要占 [1]，那 [2] 就得是一条合法后继（通常是 assistant）。对齐到 assistant 边界，是为了不把一次"工具调用 / 工具返回"从中劈开。

summary 消息：一段可见的"我做了摘要"

压缩的产物是一条特殊消息。在 v3 路径里，它由 letta/system.py::package_summarize_message_no_counts 打包成一个 type=system_alert 的 JSON 事件，正文写明"开头 N 条已隐藏，以下是它们的摘要"（带"第几条/共几条"计数的老版 package_summarize_message 只留给 legacy agent.py）。

letta/system.py · letta/services/summarizer/compact.py摘要消息 + role 取值（简化）
# v3 路径用的打包器：不带 "第几条/共几条" 计数，故名 _no_counts
def package_summarize_message_no_counts(summary, timezone, compaction_stats=None, mode=None):
    # 滑窗模式：从 stats 算出驱逐了几条
    num_evicted = compaction_stats["messages_count_before"] - compaction_stats["messages_count_after"]
    context_message = (
        f"Note: {num_evicted} messages from the beginning of the conversation "
        f"have been hidden ... due to memory constraints.\n"
        f"The following is a summary of the previous messages:\n {summary}"
    )
    return json_dumps({"type": "system_alert", "message": context_message, ...})

# 摘要以什么 role 入库，取决于开关 use_summary_role（源自 include_compaction_messages）
if use_summary_role:                 # 开 -> 一等公民 role=summary
    m = Message(role=MessageRole.summary, ...)
else:                                # 默认关 -> 退化成 role=user
    m = Message(role=MessageRole.user, ...)

这条摘要消息有两副面孔。对模型（始终如此）：它就摆在上下文里（紧跟系统消息之后），让模型一眼读到"前情提要"——这一半跟开关无关。对客户端（需打开开关）：agent 才额外 yield 一组结构化事件——压缩开始前抛 event_type="compaction" 的 EventMessage，完成后抛 SummaryMessage，UI 才显示一条"已压缩 N 条消息"。

还记得第 11 课的角色表吗？Message.role 里那个不起眼的 summary，正是为这一刻准备的。打开 use_summary_role，摘要就以 role=summary 一等公民的身份入库；默认关时它退化成一条 role=user 的 system_alert——内容照样在、模型照样读，只是少了那个角色标签。

⚠️一个开关，决定"可见 vs 静默"。include_compaction_messages 默认 False：此时不向客户端流式抛 compaction 事件，摘要也以 role=user 的 UserMessage 呈现并入库。打开它，才升级成"compaction 事件 + role=summary 一等公民"。但不管开关与否，摘要本身始终生成、始终被模型读到——可见性是能力，不是默认。

能压 / 不能压：核心记忆为什么压不动

压缩有个边界：它只清理对话消息，绝不动核心记忆与系统提示。原因很简单——core 是 system 的一部分，是 agent"此刻是谁、记着哪些事"的依据，压掉它等于让 agent 失忆。

✅ 能压：对话消息

较旧的 user / assistant / tool 消息，总结成一段摘要、换出窗外；原话仍留 recall 可搜。

compact_messages → summarize

⛔ 不能压：核心记忆 / 系统提示

第 0 条系统消息（含 core memory）是稳定前缀，summarizer 一行都不碰。

system message [0] · 原样保留

于是冒出一个尴尬情形：万一核心记忆自己就撑爆了预算呢？压缩只能驱逐对话，对话压光了也救不了一个过大的系统提示。这时 Letta 走一条特判：_check_for_system_prompt_overflow。

⚠️系统提示溢出，是一条专门的"压不动"分支。压完仍超阈值、且一数发现"光第 0 条系统消息就 ≥ 整个窗口"——这不是对话太长，是 core/系统提示太大。Letta 停下并抛 SystemPromptTokenExceededError，停止原因记为 context_window_overflow_in_system_prompt，提示你"该瘦身核心记忆，而不是再压对话"。

这也回扣了第 7、8 课的告诫：core 有字符上限（CORE_MEMORY_BLOCK_CHAR_LIMIT），是稀缺资源。压缩能化解"对话型"记忆压力，却化解不了"你往 core 里塞太多"——后者只能靠你自己克制。

一个真实场景：聊了两小时还没爆窗

把机制串成一个故事。你和一个客服 agent 聊了两个钟头，几十轮来回，外加好几次工具调用。换成"把全部历史塞进 prompt"的朴素做法，早就撞窗报错了。

但这个 agent 一直好端端的。秘密就是那条 90% 水位线：每当在窗 token 逼近上限，它就悄悄做一次压缩——把前一个钟头的闲聊总结成一两段摘要，原始几十条消息退到窗外。

📌第 40 轮时，UI 上闪过一行"已压缩 38 条消息"——那就是 compaction 事件。摘要进了上下文，前情没丢；你随口提起"开头说的那个订单号"，它还能用 conversation_search（第 11 课）从 recall 里把原话捞回来。

这就是"记忆压力"被管住的样子：对话可以无限长，窗口却始终在预算内转。摘要扛起"前情提要"，recall 兜底"原话可查"，两者一配合，agent 既不爆窗、又不真正失忆。

💡 设计亮点
"记忆压力"被当成一个操作系统问题来解——这正是 MemGPT 论文"队列管理器"的落地。操作系统怎么应对内存不够？它不会崩，而是换页：把不常用的页换出到磁盘，腾出物理内存。Letta 对上下文窗口做的是同一件事：到 ~90% 水位，就把较旧的对话递归地总结成一段摘要（旧摘要 + 新消息再压一层，越久越浓缩），把原话"换出"到窗外的 recall。三个设计选择让它格外像 OS：其一，有水位线——不等真满（撞 OOM）才动手，留 10% 余量主动触发；其二，换页可观测——压缩能抛一个 compaction 事件、把摘要以 role=summary 一等公民入库（由 include_compaction_messages 开关控制，默认关时降为静默 + role=user，但摘要照样生成、照样被读到），让"记忆操作"可以对 agent 与用户透明，而不是黑箱丢数据；其三，有一块"锁定内存"不可换出——核心记忆与系统提示是 agent 的身份，summarizer 永不触碰，于是必须另设一条系统提示溢出特判去处理"锁定区自己就爆了"的极端情况。把这三点合起来，你看到的不是一个"聊久了就失忆"的聊天机器人，而是一个像内核一样主动管理自身内存压力的系统——这也是 MemGPT 那篇论文最核心的隐喻：给 LLM 配一个虚拟内存管理器，让有限的上下文窗口，撑起近乎无限的对话。
⚠️ 常见误区
摘要是有损的，而且压缩救不了"核心记忆自己太大"。两件事都容易被想当然，得分开记清。第一，压缩 = 有损总结，不是无损归档。一段对话被总结成"用户咨询了退货并给了订单号"，那些具体措辞、语气、边角细节就从在窗视野里消失了。好在它们没被删——原始 Message 仍躺在 recall 里，conversation_search 能捞回来。但你不能假设"摘要里没有 = 模型还记得清"：摘要里没有，就意味着模型当下看不到、得主动去搜才行。所以别把压缩当成"免费无限记忆"——它是用"细节精度"换"窗口空间"的一笔交易。第二，压缩只驱逐对话、不动 core。如果你把一大堆东西塞进核心记忆，把系统提示撑到逼近甚至超过整个窗口，那么压多少对话都没用——summarizer 根本不碰 core。这时触发的是 _check_for_system_prompt_overflow 那条特判：直接报 SystemPromptTokenExceededError，停止原因 context_window_overflow_in_system_prompt。它在提醒你："问题不在对话，在你的 core 太胖，去精简记忆块、或换更大的窗口。"记住这条边界，你就不会一边往 core 狂塞、一边纳闷"为什么压缩不管用"。
🔬 落到代码
整条压缩链路的源码坐标。阈值：letta/services/summarizer/thresholds.py::get_compaction_trigger_threshold = context_window × SUMMARIZATION_TRIGGER_MULTIPLIER，后者在 letta/constants.py 为 0.9。触发与动作：letta/agents/letta_agent_v3.py 里，_step 把 context_token_estimate（取自 LLM 返回的 usage.total_tokens）跟阈值比，超了就调 compact，并抛 _create_compaction_event_message。压缩实现：letta/services/summarizer/compact.py::compact_messages，选段与递归摘要在 summarizer.py::Summarizer（_partial_evict_buffer_summarization，默认 partial_evict_summarizer_percentage=0.30）。摘要消息：v3 路径打包用 letta/system.py::package_summarize_message_no_counts（打成 system_alert；带计数的老版只留给 legacy agent.py），角色取值 letta/schemas/enums.py::MessageRole.summary（默认关时为 user）。溢出特判：letta/agents/letta_agent_v3.py::_check_for_system_prompt_overflow 与 compact.py 末尾的同款检查，触发 SystemPromptTokenExceededError（letta/errors.py）。顺着这串符号读，你能把"从超阈值到摘要落库"完整走通。

再挖深一点

递归摘要到底怎么做：摘要也会被再摘要

示例：第一次压缩，把第 1–30 条总结成摘要 S1，放到 [1]。又聊了 30 轮、再次逼近 90%，第二次压缩会把"S1 + 第 31–60 条"一起总结成 S2。

为什么这样设计：历史无限增长，但摘要长度大体可控——越久远的内容被反复压缩、层层凝练，近期的保留更多细节。这正是 MemGPT 论文"递归摘要"的核心。

源码在哪：summarizer.py::_partial_evict_buffer_summarization——它把摘要消息固定放在 [1]，下一轮该位置又被纳入新的待总结区间。

代价：越老的信息精度损失越大。但"原话仍在 recall"是兜底——真要细节，还能搜回来。

为什么核心记忆压不动：系统提示溢出特判

示例：你往 persona / human 块塞了几万字，系统提示本身就快顶满窗口。这时哪怕把对话全压光，也腾不出空间。

为什么这样设计：core 是 agent 的身份与即时记忆，必须始终在窗、且逐字稳定（还关系到第 5 课的 prefix cache）。让 summarizer 去压它，等于让 agent 一边说话一边失忆——绝不允许。

源码在哪：compact.py 压完会复核：若 compacted_messages[0]（系统消息）单独就 ≥ 窗口，抛 SystemPromptTokenExceededError；letta_agent_v3.py::_check_for_system_prompt_overflow 是同款检查，停止原因 context_window_overflow_in_system_prompt。

给你的信号：遇到它，别再想"多压点对话"，而要去精简核心记忆（回看第 8 课的 limit）或换更大的窗口。

可见的 compaction 事件：客户端怎么知道压过了

示例：UI 上出现一行"压缩中…"，紧接着一条"已将 38 条消息总结为摘要"。

为什么这样设计：记忆压缩会改变上下文，偷偷做会让用户困惑（"它怎么忘了刚才的细节？"）。抛成显式事件，行为可观测、可解释。

源码在哪：letta_agent_v3.py::_create_compaction_event_message 产出 event_type="compaction" 的 EventMessage（压缩开始前抛）；_create_summary_result_message 产出 SummaryMessage（压缩完成后抛）。两者都受 include_compaction_messages 控制（默认 False → 都不流式抛、摘要降级为 UserMessage）。

对比：很多框架"截断历史"是无声的；Letta 把它做成一等事件，是"记忆操作要透明"这一哲学的延续。

和第 5 课"度量"如何接上"动手"

示例：第 5 课用 ContextWindowOverview 把窗口拆成 system / core / 工具 / 消息各占多少 token——那是"体检报告"。第 12 课的压缩，是读着报告动手治疗。

为什么这样设计："能量出来"和"能压下去"是两件事。光有账本不压，窗口照样爆；光会压不会量，又不知何时该动手。两半合起来才是闭环。

源码在哪：度量 letta/schemas/memory.py::ContextWindowOverview（第 5 课）；判定 + 动手 letta_agent_v3.py（context_token_estimate 比 get_compaction_trigger_threshold，超则 compact）。

一句话：第 5 课"量得出"，第 12 课"压得动"——同一条闭环的前后两半。

收尾：记忆系统至此闭环

这一课你拿下了让整个记忆系统"转得动"的那道阀门：到 90% 阈值就 compact，用滑窗递归摘要把较旧对话压成一条 role=summary 消息，过程可观测，唯独核心记忆压不动、另有溢出特判。

What happens when the window fills up? The answer is compaction: summarize the older conversation into a compact summary and evict the verbatim messages. 🔌 Analogy
Picture "memory pressure" as a desk filling up. You're spreading papers across it to get work done, but the surface is finite — the stack grows until there's no room left to set down a pen. The smart move isn't to throw the old papers away (what if you need them?), but to spend a minute reading the older stack and writing a one-page "meeting minutes": the key points land on the minutes, the thick original stack goes into a drawer. Now most of the desk is clear, you can spread new papers again, and that minutes page stays in front of you reminding you "here's where we were." The originals weren't destroyed — you can pull them from the drawer to double-check; but for day-to-day work, the compressed minutes is enough. Letta's context compaction does exactly this — except "the older stack" is old conversation messages, the "minutes" is an auto-generated summary, and what decides "when to clear the desk" is a watermark drawn at 90%. Remember this desk and every mechanism below has a place to land.
🌍 Big picture
Grab this lesson in one line: when in-window tokens approach 90% of the limit, Letta compacts the older conversation into a summary and evicts the originals to free space — treating "memory pressure" as an operating-system problem. The trigger is a watermark: after each model call it records usage (context_token_estimate), and once that exceeds the context_window × 0.9 threshold (get_compaction_trigger_threshold), it calls compact. The act is a sliding-window summary: hand an older slice of messages to a summarizer, recursively condense it into one role=summary message inserted into context, while the originals drop out of the window but stay searchable in recall. The whole thing can also (when enabled) emit a visible event so the client sees "I just compacted." The one thing it can't touch is core memory and the system prompt — they're part of system, so the summarizer only clears conversation; hence a separate system-prompt overflow path as a backstop.

First, connect this lesson back to Lesson 5

Lesson 5 drew a loop: measure → decide → compact → rebuild prefix. The first two steps (count who holds how many tokens, decide whether you're near the threshold) were covered there; this lesson picks up at step three, "compact" — how it actually acts, and what happens afterward.

🧠One-line placement: Lesson 5 is "you can measure it" (the ContextWindowOverview ledger); Lesson 12 is "you can compact it" (the compact scissors). Two halves of one loop; this lesson completes the back half.

Why is compaction unavoidable? Because recall (Lesson 11) guarantees the conversation is never lost, but doesn't solve "won't fit in-window." Messages only grow; eventually they blow the budget. Compaction is the valve that keeps the window turning — it doesn't delete history, it just swaps an older slice for a shorter summary.

Window nearly full (~90%): compress an older stack of messages into one "meeting minutes," freeing space to keep going; originals drop out of the window but stay searchable

The 90% threshold: when compaction starts

Compaction doesn't wait for the window to be "actually full" — that easily slams into a "prompt too long" error. Letta keeps a 10% margin: it triggers at 90% of the limit. That threshold comes from get_compaction_trigger_threshold, equal to context_window × 0.9.

usage 0% · chat freely 70% · still roomy 90% · trigger threshold
action near 90% → compact: compress older messages into a summary evict originals · insert summary
result usage drops → keep chatting; originals stay searchable in recall
letta/services/summarizer/thresholds.py · letta/constants.pycompaction trigger threshold (simplified)
# trigger threshold = context window x 0.9 (10% margin, avoid "prompt too long")
SUMMARIZATION_TRIGGER_MULTIPLIER = 0.9   # letta/constants.py

def get_compaction_trigger_threshold(llm_config, *, force_proactive=False):
    # model-agnostic: always limit x 0.9 (force_proactive doesn't change the result today)
    return int(llm_config.context_window * SUMMARIZATION_TRIGGER_MULTIPLIER)
📌The threshold is a fraction of the window, not a hard-coded number: an 8k window triggers around 7.2k, a 128k window around 115k. However big the window, 90% still compacts — a larger window only delays "when," it doesn't cancel it.

The trigger point: usage vs threshold, right in the agent loop

Where does the threshold get used? In the main loop of letta/agents/letta_agent_v3.py. For speed, the runtime doesn't recompute the whole ledger each step — it records the token usage each model call reports and compares that estimate to the threshold.

1

record usage

As the model returns, store the usage.total_tokens it reports into context_token_estimate.

2

compare threshold

estimate > get_compaction_trigger_threshold (×0.9)?

3

optional: emit event

when enabled, first yield a compaction event telling the client "about to compact."

4

do the compaction

call self.compact(messages, ...) to swap older messages for a summary.

Key detail: what gets compared isn't a "ledger-precise" value but the usage.total_tokens the last model call reported itself. That's like having the model weigh itself each turn — no recompute, yet accurate enough. Exceed the threshold and it enters the compaction branch.

💡Two trigger paths, one destination. One is the post-step watermark check (this step done, usage past 90% → compact, trigger="post_step_context_check"); the other is the hit-the-wall backstop (the model raises ContextWindowExceededError, compact-and-retry in place, trigger="context_window_exceeded"). Both funnel into the same compact.

compact_messages: compress old messages into a summary

self.compact(...) is just a thin shell; the real work is in letta/services/summarizer/compact.py::compact_messages. It takes the current in-window messages, hands them to the summarizer, and produces three things: a summary message, the compacted message list, and the summary text.

letta/services/summarizer/compact.pycompact_messages (pseudocode)
async def compact_messages(..., messages, use_summary_role=True, trigger_threshold=None, ...):
    # 1) summarize away an older slice of messages (sliding-window / recursive)
    summary, compacted_messages = await summarize(messages)

    # 2) re-count tokens, confirm we actually dropped below the threshold
    context_token_estimate = await count_tokens_with_tools(messages=compacted_messages, ...)

    # 3) if still over AND the culprit is message[0] (system) -> overflow special case
    if context_token_estimate >= trigger_threshold:
        if tokens_of(compacted_messages[0]) >= llm_config.context_window:
            raise SystemPromptTokenExceededError(...)   # core memory/system prompt too big

    return CompactResult(summary_message, compacted_messages, summary, ...)

Note step 3: after compacting it re-counts tokens to verify (still over → it first tries a full "summarize-all" fallback). If even that can't save it, and a check shows "the culprit is message[0], the system message itself" — then it's not that the conversation is too long, it's that core memory/system prompt blew the budget. That fork is the "system-prompt overflow" we'll cover next.

Sliding-window summary: which slice gets compressed, and how

How does "compaction" pick messages? Via the Summarizer (summarizer.py). It has several modes; the common one is sliding-window / partial-evict. The core idea in one line: keep the recent slice, summarize away the earlier slice.

1

compute a cut line

partial_evict_summarizer_percentage (default 0.30) is the kept ratio: keep the most recent ~30%, summarize the older ~70%.

2

align to a boundary

from the cut point, find the first assistant message so role order stays valid after re-insert.

3

summarize the evicted slice

hand [1 : that assistant) to the summarizer LLM, producing one summary.

4

replace index 1

put the summary back at position 1 (position 0 is always the system message); originals leave the window.

This is the MemGPT paper's recursive summary: the summary is pinned at [1], and the next time you compact, it gets summarized again together with the newer messages. History snowballs into an ever-more-condensed single summary — the older it is, the more distilled.

Why "find an assistant after the cut point"? Because the summary takes [1], so [2] must be a valid successor (typically an assistant). Aligning to an assistant boundary avoids splitting a tool-call / tool-response pair down the middle.

The summary message: a visible "I made a summary"

The product of compaction is a special message. On the v3 path, letta/system.py::package_summarize_message_no_counts packs it into a type=system_alert JSON event whose body says "N messages from the start were hidden; here's their summary" (the with-counts package_summarize_message is reserved for the legacy agent.py).

letta/system.py · letta/services/summarizer/compact.pysummary message + role choice (simplified)
# the v3 packer: drops the "of N" counts, hence _no_counts
def package_summarize_message_no_counts(summary, timezone, compaction_stats=None, mode=None):
    # sliding-window: derive how many were evicted from stats
    num_evicted = compaction_stats["messages_count_before"] - compaction_stats["messages_count_after"]
    context_message = (
        f"Note: {num_evicted} messages from the beginning of the conversation "
        f"have been hidden ... due to memory constraints.\n"
        f"The following is a summary of the previous messages:\n {summary}"
    )
    return json_dumps({"type": "system_alert", "message": context_message, ...})

# which role the summary is stored under depends on use_summary_role (from include_compaction_messages)
if use_summary_role:                 # on -> first-class role=summary
    m = Message(role=MessageRole.summary, ...)
else:                                # default off -> falls back to role=user
    m = Message(role=MessageRole.user, ...)

This summary message has two faces. To the model (always): it sits right in context (just after the system message), letting the model read the "story so far" at a glance — this half is switch-independent. To the client (needs the switch): only then does the agent also yield structured events — before compaction an EventMessage with event_type="compaction", and after a SummaryMessage — so the UI can show "compacted N messages."

Remember Lesson 11's role table? That unassuming summary in Message.role was made for this moment. Turn on use_summary_role and the summary is stored as a first-class role=summary citizen; with the default off it falls back to a role=user system_alert — the content is still there and the model still reads it, just without that role label.

⚠️One switch decides "visible vs silent." include_compaction_messages defaults to False: no compaction events stream to the client, and the summary surfaces and persists as a role=user UserMessage. Turn it on to upgrade to "compaction event + first-class role=summary." But switch or not, the summary itself is always created and always read by the model — visibility is a capability, not the default.

Compressible / not: why core memory can't be compacted

Compaction has a boundary: it only clears conversation messages and never touches core memory or the system prompt. The reason is simple — core is part of system, the basis for "who the agent is right now and what it remembers"; compacting it amounts to giving the agent amnesia.

✅ Compressible: conversation

older user / assistant / tool messages — summarized and evicted; originals stay searchable in recall.

compact_messages → summarize

⛔ Not: core memory / system prompt

message[0], the system message (with core memory), is the stable prefix; the summarizer never touches a line.

system message [0] · kept verbatim

So an awkward case arises: what if core memory itself blows the budget? Compaction can only evict conversation, and clearing all conversation still won't save an oversized system prompt. Here Letta takes a special case: _check_for_system_prompt_overflow.

⚠️System-prompt overflow is a dedicated "can't compact" branch. Still over threshold after compacting, and a count shows "message[0] alone ≥ the whole window" — that's not a too-long conversation, it's too-big core/system prompt. Letta stops and raises SystemPromptTokenExceededError, stop reason context_window_overflow_in_system_prompt, telling you "slim down core memory, don't compact more conversation."

This circles back to Lessons 7 and 8: core has a char limit (CORE_MEMORY_BLOCK_CHAR_LIMIT) and is a scarce resource. Compaction relieves "conversational" memory pressure, but it can't relieve "you stuffed too much into core" — that one is on your own restraint.

A real scenario: two hours of chat, window never blows

Let's thread the mechanics into a story. You chat with a support agent for two hours, dozens of turns, plus several tool calls. With the naive "stuff all history into the prompt" approach, it would have blown the window long ago.

But this agent stayed perfectly fine. The secret is that 90% watermark: whenever in-window tokens approach the limit, it quietly runs a compaction — summarizing the past hour of chatter into a paragraph or two, dropping the original dozens of messages out of the window.

📌At turn 40, a line flashes in the UI: "compacted 38 messages" — that's the compaction event. The summary went into context, the story so far isn't lost; if you casually mention "that order number from the start," it can still use conversation_search (Lesson 11) to pull the original back from recall.

This is what managed "memory pressure" looks like: the conversation can be arbitrarily long, yet the window always stays within budget. The summary carries the "story so far," recall backstops "originals are queryable," and together the agent neither blows the window nor truly forgets.

💡 Design highlight
"Memory pressure" is solved as an operating-system problem — this is exactly the MemGPT paper's "queue manager," realized. How does an OS handle low memory? It doesn't crash — it pages: swap rarely-used pages out to disk to free physical RAM. Letta does the same to the context window: at the ~90% watermark it recursively summarizes older conversation into one summary (old summary + new messages compressed one layer further, more condensed over time) and "swaps" the originals out to recall. Three design choices make it feel like an OS. First, there's a watermark — it acts at a 10% margin rather than waiting for a true OOM. Second, paging is observable — compaction can emit a compaction event and store the summary as a first-class role=summary message (gated by include_compaction_messages; with the default off it degrades to silent + role=user, yet the summary is still produced and still read), so the memory operation can be transparent to agent and user instead of a black box dropping data. Third, there's a "locked" region that can't be swapped — core memory and the system prompt are the agent's identity, the summarizer never touches them, hence a separate system-prompt overflow special case to handle "the locked region itself overflowed." Put these together and you see not a chatbot that forgets after a while, but a system that, like a kernel, actively manages its own memory pressure — the core metaphor of the MemGPT paper: give the LLM a virtual memory manager so a finite context window can sustain a near-infinite conversation.
⚠️ Common pitfalls
The summary is lossy, and compaction can't save "core memory itself is too big." Both are easy to take for granted; keep them straight. First, compaction = lossy summary, not lossless archive. A stretch of conversation summarized to "the user asked about a return and gave an order number" makes the exact wording, tone, and edge details vanish from the in-window view. The good news: they weren't deleted — the original Message rows still sit in recall, and conversation_search can fetch them. But you can't assume "not in the summary = the model still remembers it": not in the summary means the model can't see it right now and must actively search. So don't treat compaction as "free infinite memory" — it's a trade of "detail precision" for "window space." Second, compaction only evicts conversation, not core. If you stuff a pile into core memory and push the system prompt toward or past the whole window, then no amount of compacting conversation helps — the summarizer never touches core. What triggers then is the _check_for_system_prompt_overflow special case: a direct SystemPromptTokenExceededError, stop reason context_window_overflow_in_system_prompt. It's telling you: "the problem isn't the conversation, your core is too fat — trim the memory blocks, or use a bigger window." Remember this boundary and you won't keep cramming core while wondering "why isn't compaction working."

Going deeper

How recursive summary works: summaries get summarized too

Example: first compaction summarizes messages 1–30 into summary S1, placed at [1]. After 30 more turns and nearing 90% again, the second compaction summarizes "S1 + messages 31–60" together into S2.

Why designed this way: history grows without bound, but summary length stays roughly bounded — older content is repeatedly compressed, layer by layer, while recent content keeps more detail. This is the heart of the MemGPT paper's "recursive summarization."

Where in source: summarizer.py::_partial_evict_buffer_summarization — it pins the summary message at [1], and the next round folds that position into the new range to summarize.

The cost: the older the info, the more precision lost. But "originals stay in recall" is the backstop — need the detail, search it back.

Why core memory can't be compacted: the system-prompt overflow case

Example: you stuff tens of thousands of chars into persona / human blocks, and the system prompt alone nearly fills the window. Now even compacting all conversation frees no space.

Why designed this way: core is the agent's identity and immediate memory — it must always be in-window and byte-stable (it also ties to Lesson 5's prefix cache). Letting the summarizer compress it would make the agent forget mid-sentence — never allowed.

Where in source: compact.py rechecks after compacting: if compacted_messages[0] (the system message) alone is ≥ the window, raise SystemPromptTokenExceededError; letta_agent_v3.py::_check_for_system_prompt_overflow is the same check, stop reason context_window_overflow_in_system_prompt.

The signal to you: when you hit it, stop thinking "compact more conversation" and instead trim core memory (revisit Lesson 8's limit) or use a bigger window.

The visible compaction event: how the client knows it happened

Example: a line appears in the UI, "compacting…", followed by "summarized 38 messages into a summary."

Why designed this way: compaction changes the context, and doing it silently confuses users ("why did it forget that detail?"). Emitted as an explicit event, the behavior is observable and explainable.

Where in source: letta_agent_v3.py::_create_compaction_event_message yields an EventMessage with event_type="compaction" (before compaction); _create_summary_result_message yields a SummaryMessage (after). Both are gated by include_compaction_messages (default False → neither streams, and the summary degrades to a UserMessage).

Contrast: many frameworks "truncate history" silently; Letta makes it a first-class event — a continuation of the "memory operations should be transparent" philosophy.

How Lesson 5's "measure" connects to "act"

Example: Lesson 5 used ContextWindowOverview to break the window into how many tokens system / core / tools / messages each hold — that's the "checkup report." Lesson 12's compaction is reading the report and treating the patient.

Why designed this way: "can measure" and "can compact" are two things. Measure without compacting and the window still blows; compact without measuring and you don't know when to act. The two halves together make the loop.

Where in source: measure letta/schemas/memory.py::ContextWindowOverview (Lesson 5); decide + act letta_agent_v3.py (context_token_estimate vs get_compaction_trigger_threshold, over it compact).

In one line: Lesson 5 "you can measure," Lesson 12 "you can compact" — the front and back halves of one loop.

Wrap-up: the memory system is now a closed loop

This lesson got you the valve that keeps the whole memory system turning: at the 90% threshold, compact; use sliding-window recursive summary to compress older conversation into one role=summary message; the process is observable; only core memory can't be compacted, with its own overflow special case.

Thread all of Part 3 together — this is the capstone of Lesson 7's three-tier map:

The memory system (07–12): three tiers + the compaction valve that keeps it turning
07 three tiers · 08 memory blocks · 09 self-editing core · 10 archival vectors · 11 recall history → 12 compaction valve

Replay block by block: core lets the agent rewrite who it is (07–09), archival lets it accumulate long-term knowledge (10), recall lets it remember every line (11), and compaction (12) keeps all of it running inside a finite window. With all four in place, the memory system truly closes the loop.

👉Bridging back and forward: Lesson 5 posed the root constraint "the window is finite" and gave you "measure"; Part 3 (07–12) tells the whole "memory system" answer — and compaction is the last piece that fights the pressure.

Memorize one chain on your way out: record usage each turn → past the 90% threshold → emit a compaction event → compact via sliding-window recursive summary → insert role=summary, originals retreat to recall → only core can't be compacted; if it overflows, take the system-prompt overflow special case. Hold this chain and you've truly grasped "context compaction & memory pressure."

✅ Key points
  • 90% threshold trigger: get_compaction_trigger_threshold = context_window × 0.9 (SUMMARIZATION_TRIGGER_MULTIPLIER in constants.py); a 10% margin to compact proactively, not at the wall.
  • Trigger in the agent loop: letta_agent_v3.py compares context_token_estimate (= the LLM-reported usage.total_tokens) to the threshold; over it calls compact.
  • compact_messages does the work: summarizer/compact.py summarizes away older messages; slice selection + recursive summary in Summarizer (partial-evict, default 0.30), summary pinned at [1].
  • The summary message: the v3 path uses system.py::package_summarize_message_no_counts to pack a system_alert; turn on include_compaction_messages to store it under role=summary and stream a compaction event / SummaryMessage (default off → role=user, silent).
  • Core memory can't be compacted: the summarizer only evicts conversation, never core/system prompt; if core overflows → _check_for_system_prompt_overflow → SystemPromptTokenExceededError.
  • Compaction is lossy but non-deleting: the summary drops detail, originals stay in recall via conversation_search; this is the MemGPT "queue manager," realized.
