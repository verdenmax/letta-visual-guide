工具规则 Tool RulesTool rules

第 14 课拆 V3 循环时，_decide_continuation 里闪过一个判官 is_terminal_tool，第 15 课又点出另两个——has_children_tools、is_continue_tool；第 15 课结尾还留了一句话：这些"工具规则"到底是谁在背后掌舵，留到第 16 课讲。这一课，就把这笔账还清。

这是第四部分的收尾课，主题只有一句：把"agent 该怎么用工具"从模型的自觉，变成一台声明式、可强制执行的状态机。三件事带走：9 种规则类型、一个每步现算"合法下一步"的求解器，以及最妙的一招——模型若挑了非法工具，框架不执行它，而是合成一条错误喂回去让它自己改。

🚦 生活类比
把工具规则想成一座城市的交通规则。run_first 是"出厂必须先上这条主路"——开口回话前先去查资料；exit_loop 是"终点站，到了就下车"——调了 send_message 这一轮就收尾；constrain_child_tools 是"这个路口只许拐进这几条岔路"——调完 规划 工具，下一步只能走 执行 那几个孩子；required_before_exit 是"离厂前必须先打卡"——没调过结算工具，不准退出循环。关键在于：红绿灯不靠司机自觉，它装在路口、由市政强制——闯了就是闯了，灯不会因为你赶时间就变绿。工具规则做的是同一件事：把"该怎么用工具"从司机的素质，搬成路口的硬件。司机再莽撞，红灯也拦得住。
🌍 宏观理解
一句话抓住本课：工具规则是一套声明式约束，让框架而不是模型来拍板"这一步能调哪些工具"。它由三块拼成。其一，9 种规则类型（schemas/tool_rule.py 的九个子类，各 pin 一个 Literal type，按 type 区分的判别联合）——分别管"起点 / 终点 / 下一步 / 限频 / 必调 / 审批"。其二，一个求解器现算合法集：helpers/tool_rule_solver.py::ToolRulesSolver 把规则分桶，每一步算一次 get_allowed_tool_names——首步若配了 init 规则就强制只返回 init 工具，否则把 child / parent 规则各自的"合法集"求交、再 & 当前可用工具。其三，也是最妙的：模型若仍挑了非法工具，框架不执行它，而是 agents/helpers.py::_build_rule_violation_result 合成一条 [ToolConstraintError] 当作"工具返回"喂回模型，让它自己改。约束因此两头都管：事前裁掉发给 LLM 的工具 schema，事后再用合成错误兜底。记住这三块的次序——规则声明 → 求解器现算 → 违规兜底——下面三节正是按它逐层展开。

一、九种工具规则：声明式约束的"字母表"

先认全这套"字母表"。九种规则都住在 schemas/tool_rule.py，是一组判别联合——每个子类 pin 死一个 Literal type 字符串，靠这个 type 把它们区分开。枚举名单在 schemas/enums.py::ToolRuleType。

class InitToolRule(BaseToolRule):                # run_first：必须第一个调
    type: Literal["run_first"] = "run_first"
    args: dict | None = None                    # 可选：首步预填的参数

class TerminalToolRule(BaseToolRule):            # exit_loop：调到就结束循环

class ChildToolRule(BaseToolRule):               # constrain_child_tools：限制下一步
    type: Literal["constrain_child_tools"] = "constrain_child_tools"

顺带点出一个"超纲"能力：规则不止能限制工具，还能预填参数——InitToolRule.args 可以给首步工具预先塞好实参，ChildToolRule.child_arg_nodes 还能约束孩子工具的参数取值。基础用法之外，这层细节让规则比"白名单"更有表达力：它不只说"能调谁"，还能说"调它时大致怎么填"。

🧠九个子类共一个基类 BaseToolRule，它提供两样东西：方法 get_valid_tools(tool_call_history, available_tools, last_function_response)（这条规则此刻"放行哪些工具"，返回一个集合），以及属性 requires_force_tool_call（要不要逼模型这一步必须调工具）。求解器就是靠这套统一接口，把九种规则一视同仁地算到一起。

二、ToolRulesSolver：每一步现算"合法下一步"

规则是静态声明，真正把它们"跑起来"的是 ToolRulesSolver。它挂在 V2 基类上——self.tool_rules_solver = ToolRulesSolver(tool_rules=agent_state.tool_rules)——V3 直接继承复用。构造时 model_post_init 按 isinstance 把规则分桶。

核心方法是 get_allowed_tool_names(available_tools, error_on_empty=True, last_function_response=None)。它只有两条岔路：要么是"首步强制走 init 工具"，要么是"把相关规则的合法集求交"。求交的妙处下面专门讲——这里先记住：合法集是每一步现算的，不是一次性定死的。因为 tool_call_history 每调一次工具就被 register_tool_call 更新，孩子规则、限频规则下一步看到的"合法集"也随之变化。

⚠️最容易记反的一点：get_allowed_tool_names 只对 child + parent 规则求交来缩小工具集。terminal / continue / required_before_exit / requires_approval 这几桶根本不进交集——它们不缩小"能调哪些"，而是在循环里左右"调了之后怎么走"。solver 的文档字符串原话就提醒：Continue / Terminal / RequiredBeforeExit 规则在 agent 循环中生效，不用来限制工具。
letta/helpers/tool_rule_solver.pyget_allowed_tool_names 核心（简化）
def get_allowed_tool_names(self, available_tools, error_on_empty=True,
                           last_function_response=None) -> list:
    # 第一步且配了 init 规则 -> 强制只返回 init 工具
    if not self.tool_call_history and self.init_tool_rules:
        return [r.tool_name for r in self.init_tool_rules]

    # 否则：把 child + parent 规则各自的"合法集"求交集
    relevant = self.child_based_tool_rules + self.parent_tool_rules
    if not relevant:
        return list(available_tools)
    allowed = set.intersection(*[
        r.get_valid_tools(self.tool_call_history, available_tools,
                           last_function_response)
        for r in relevant
    ])
    return list(allowed & set(available_tools))   # 再 & 一道可用工具

算出合法集后，V3 的 _get_valid_tools 拿它去裁剪真正发给 LLM 的那批工具 JSON schema——非法工具压根不出现在模型眼前。同时 should_force_tool_call() 决定这一步用 forced（逼模型必须调工具，比如首步 init）还是 auto（让模型自由发挥），终止工具的处理则走 runtime_override_tool_json_schema（第 15 课见过的那道改写关）。

🧠合法集是动态的：每执行一个工具，solver 就用 register_tool_call 把它记进 tool_call_history，下一步的 child / parent / 限频规则看到的"合法集"随之收窄。所以循环每一步的"可调范围"都是现算现裁，而非开局定死——这也是 get_valid_tools 必须把"调用历史"和"上次返回"都传进去的原因。
🚦路口 · solver → 🧰此刻只有这几个工具亮绿灯 红灯的，schema 里直接裁掉
求解器每一步现算"合法集"，等于在路口给工具亮灯——红灯的工具模型根本看不到（schema 里被裁掉），从源头少犯错

三、违规不执行：把错误"合成"出来喂回模型

裁了 schema，模型就一定听话吗？不一定——它仍可能无视约束、硬挑一个非法工具（旧 prompt、幻觉、或干脆不读 schema）。这时第二道防线登场：违规的工具不会被执行。

判定这一步发生在 letta_agent_v3.py::_handle_ai_response：它先算 tool_rule_violated = name not in valid_tool_names。若违规，就跳过执行，转而调 _build_rule_violation_result 造一条"假装是工具返回"的错误消息。模型读到这条消息，就明白自己挑错了、该从哪几个里重挑。

👉一个值得玩味的分工：那句 [ToolConstraintError] Cannot call X, valid tools include: [...] 的错误字符串，是在 agent 这一层（agents/helpers.py）拼出来的，不在 solver 里。solver 只通过 guess_rule_violation 额外供一组提示（猜猜你大概违反了哪条规则），拼装与回喂是 agent 的活。职责切得很干净：solver 管"算合法集"，agent 管"违规了怎么向模型解释"。
letta/agents/helpers.py_build_rule_violation_result：违规合成一条错误（简化）
def _build_rule_violation_result(tool_name, valid_tools, tool_rules_solver):
    # 注意：错误字符串在 agent 这层拼，不在 solver 里
    hint = tool_rules_solver.guess_rule_violation(tool_name)
    msg = (f"[ToolConstraintError] Cannot call {tool_name}, "
           f"valid tools include: {valid_tools}.")
    if hint:
        msg += f" Hint: {hint}"
    # 不执行该工具，只把这条"错误"当作它的返回喂回模型
    return ToolExecutionResult(status="error", func_return=msg)

这一招为什么优雅？因为它把"违规"从一次崩溃变成一次教学。更妙的是它和第 14 课严丝合缝：违规仍然算"这一轮调了工具"——于是 _decide_continuation 那条"调了工具就继续"判据照样成立，循环自然又走一圈，正好把改正的机会递给模型。第 14 课那个敢于"极简"的循环，底气正来自这里。

四、规则怎么左右"继续 / 停止"（接第 14 课）

同一个 solver，除了"算合法集"，还在循环里决定续 / 停：终止 / 孩子 / 继续 / 必调这几条在第 14 课的 _decide_continuation 里查；审批那条则在 _handle_ai_response 里直接拦下。这几桶不缩小工具集的规则，作用就体现在这里。

规则solver 方法在循环里的效果
exit_loop（终止）is_terminal_tool调到 → 停，停止原因 tool_rule
constrain_child / continuehas_children_tools / is_continue_tool命中 → 继续，强制再走一圈

单说 requires_approval 这条审批闸口：命中时循环不执行该工具，而是停下来，停止原因记 StopReasonType.requires_approval，并持久化一条由 create_approval_request_message_from_llm_response(...) 造的"审批请求"消息。等人点了批准，下一次进来是 is_approval_response，它绕过违规检查直接放行——这正接上第 15 课说的"工具规则可以覆盖心跳标志"，也对上第 14 课列的停止原因。

✅把第 14、15、16 课串起来：第 14 课的循环之所以敢只用"调了工具就继续"这么简单的判据，第 15 课之所以敢整个删掉 request_heartbeat——因为工具规则在背后兜底。终止、必调、审批这些"非黑即白"的决定，被声明式地焊进了规则，循环本身就能保持极简。
💡 设计亮点
工具规则把"agent 该怎么用工具"从模型的自觉，升级成一台声明式、可强制执行的状态机——里头藏着两个漂亮设计。第一，约束是"算"出来的：每一步 ToolRulesSolver 把所有相关规则的合法集求交，得到"此刻只有这几个被允许"，再据此裁剪发给 LLM 的工具 schema——非法工具压根不进模型视野，从源头减少误选。第二，违规是一次教学，而非一次崩溃：万一模型仍挑了非法工具，框架不执行它，而是返回 [ToolConstraintError] Cannot call X, valid tools include: [...] 当作那个工具的"返回值"，让模型读到、自己改正。约束因此两头都管——事前缩 schema、事后合成错误兜底——把一门"放飞的 LLM"调教成一个"照规格干活的执行器"。这也正是第 14 课那个极简循环（"调了工具就继续"）敢于极简的原因：脏活累活，工具规则替它扛了。
🔬 落到代码
四个锚点，一条链。九个子类在 schemas/tool_rule.py（各 pin 一个 Literal type，按 type 区分的判别联合），枚举名单在 schemas/enums.py::ToolRuleType，基类接口是 BaseToolRule.get_valid_tools 与属性 requires_force_tool_call。求解器是 helpers/tool_rule_solver.py::ToolRulesSolver：model_post_init 分桶，核心 get_allowed_tool_names，外加 is_terminal_tool / has_children_tools / is_continue_tool / has_required_tools_been_called / is_requires_approval_tool / should_force_tool_call / register_tool_call / guess_rule_violation。违规合成在 agents/helpers.py::_build_rule_violation_result。接线在 V3：letta_agent_v3.py::_get_valid_tools（裁 schema）、_handle_ai_response（校验 tool_rule_violated）、_decide_continuation（读终止 / 孩子 / 继续 / 必调）；solver 挂在 V2 基类 self.tool_rules_solver = ToolRulesSolver(tool_rules=agent_state.tool_rules)。
⚠️ 常见误区
四个别记错的点。其一，违规的工具不会被执行——它被 _build_rule_violation_result 换成一条合成错误喂回模型，不是"真跑了再报错"。其二，get_allowed_tool_names 只对 child + parent 规则求交来缩工具集；terminal / continue / required / approval 不缩工具集，它们在循环里起作用。其三，枚举值是历史命名：run_first / exit_loop / constrain_child_tools…，不是 Init / Terminal / Child。其四，Letta 不做环 / 冲突检测——只有逐条 pydantic 校验（ChildToolRule.child_arg_nodes ⊆ children；ConditionalToolRule 至少一条映射）＋ agent_manager_helper.py::check_supports_structured_output 对非结构化输出模型、在配了 >1 个 init 规则时报错。规则之间会不会打架，框架不替你兜，得靠配置者自己想清楚。这也提醒一句：工具规则是把"双刃剑"——用好了能精确编排，配拧了也没有保险丝替你拦下。

再挖深一点

九种规则，各一句话配个迷你例子
  • run_first（Init）：强制首步——先 archival_memory_search 再 send_message，开口前必须先查资料。
  • exit_loop（Terminal）：调到就收尾——send_message 常被设成终止工具。
  • continue_loop（Continue）：调到就强制再走一圈，不让循环停。
  • conditional：按本工具输出挑下一步——像状态机的转移函数，child_output_mapping 把输出值映射到下一个工具。
  • constrain_child_tools（Child）：调完 plan 只能调 execute 那几个孩子，把"先规划后执行"焊死。
  • max_count_per_step：限频——防止模型在一个 step 里把同一个工具刷十遍。
  • parent_last_tool（Parent）：这些孩子只能紧跟某父工具之后调。
  • required_before_exit：退出前必须调过——比如必须先 log_result 才准结束。
  • requires_approval：调用前暂停等人批——删库、转账这类危险操作的保险。
conditional 规则怎么按"输出值"挑下一步？

示例：一个 classify 工具返回 "refund" 或 "complaint"。ConditionalToolRule 的 child_output_mapping 把 "refund" 映到 process_refund、"complaint" 映到 escalate，再用 default_child 兜底没匹配上的情况。

为什么这样设计：它把"下一步调谁"做成本工具输出的函数——等于在工具规则层内置了一台状态机，不必靠模型自己记住分支逻辑。

怎么算进合法集：它被分到 child_based_tool_rules 桶，get_valid_tools 会读 last_function_response（上一个工具的返回）来决定放行哪个孩子——这正是合法集要把"上次返回"一并传进去的原因。

源码在哪：schemas/tool_rule.py::ConditionalToolRule（pydantic 校验器要求至少一条映射）；分桶在 tool_rule_solver.py::model_post_init。

为什么"合法集"要用交集而不是并集？

示例：同时配了 ChildToolRule（调完 A 后只能调 B/C）和 MaxCountPerStepToolRule（C 一步最多 1 次，且已调过）。两条都活跃：{B,C} ∩ ({B,C} 去掉被限频的 C) = {B}。

为什么这样设计：多条规则同时生效时，"合法"必须同时满足所有规则——求交就是"它们都放行的才算数"。若用并集，会放过某条规则明明禁掉的工具，约束就形同虚设。

源码在哪：tool_rule_solver.py::get_allowed_tool_names 里 set.intersection(*[r.get_valid_tools(...) for r in child_based + parent]) 再 & available。

为什么 terminal / required 这些规则不缩小工具集？

示例：TerminalToolRule 把 send_message 标成终止工具，但并不会把它从"能调的工具"里删掉——你当然还能调它，只是调了之后循环就停。

为什么这样设计：缩小工具集（get_allowed_tool_names）只回答一个问题——"下一步法律上允许调哪些"。而终止 / 继续 / 必调 / 审批管的是"调了之后循环怎么走"，是循环里的动作，不是工具过滤器。两件事分开，逻辑才清爽。

源码在哪：只有 child_based_tool_rules + parent_tool_rules 进交集；其余桶在 _decide_continuation（第 14 课）里被读。

违规为什么"合成一条错误"，而不是直接抛异常？

示例：模型无视约束挑了非法工具。若直接 raise，整个 step 崩，用户看到一个莫名其妙的报错。

为什么这样设计：_build_rule_violation_result 把违规变成一条 status="error" 的工具返回——内容是 [ToolConstraintError] Cannot call X, valid tools include: [...]。模型读到这条"返回"，就知道挑错了、该从哪几个里重挑。循环不中断，模型自纠。

接第 14 课：违规仍然算"这一轮调了工具"——所以 _decide_continuation 判"调了工具就继续"，循环又走一圈，正好给模型改正的机会。这就是第 14 课那个极简循环敢极简的底气。

源码在哪：agents/helpers.py::_build_rule_violation_result；solver 只通过 guess_rule_violation 供一句 hint。

requires_approval 的完整流程（接第 14 课停止原因）

示例：模型要调 delete_all_records，这工具配了 requires_approval。

流程：is_requires_approval_tool(name) 为真 → 循环不执行该工具，而是停下，停止原因记 StopReasonType.requires_approval，并持久化一条 create_approval_request_message_from_llm_response(...) 造的"审批请求"消息。

恢复：人点了批准后，下一次进来是 is_approval_response，它绕过违规检查，直接放行执行。

接第 14 课：这正是第 14 课列的停止原因之一——requires_approval 让循环优雅地停在"等人"，而不是硬闯危险操作。

🚦 Analogy
Think of tool rules as a city's traffic regulations. run_first is "you must take this main road first on leaving the depot" — look something up before you open your mouth to reply; exit_loop is "this is the terminal stop, get off when you arrive" — calling send_message wraps up the round; constrain_child_tools is "at this junction you may only turn into these few side roads" — after calling the plan tool, your next step can only be those execute children; required_before_exit is "clock in before you leave the depot" — you can't exit the loop without having called the settlement tool. The key point: a traffic light doesn't rely on the driver's conscientiousness, it's mounted at the junction and enforced by the city — run a red and you've run it; the light won't turn green just because you're in a hurry. Tool rules do the same thing: they move "how to use tools" from the driver's character into hardware at the junction. However reckless the driver, a red light still stops them.
🌍 Big picture
One line to grab this lesson: tool rules are a set of declarative constraints that let the framework, not the model, decide "which tools may be called this step." It's built from three blocks. First, 9 rule types (the nine subclasses in schemas/tool_rule.py, each pinning a Literal type — a discriminated union keyed on type) — covering "start / end / next step / rate-limit / must-call / approval." Second, a solver that computes the legal set live: helpers/tool_rule_solver.py::ToolRulesSolver buckets the rules and runs get_allowed_tool_names once per step — on the first step, if init rules are configured it forces a return of only the init tools; otherwise it intersects the child / parent rules' "legal sets" and then & the currently available tools. Third, and cleverest: if the model still picks an illegal tool, the framework does not execute it but, via agents/helpers.py::_build_rule_violation_result, synthesizes a [ToolConstraintError] as the "tool return" and feeds it back for the model to fix itself. The constraint thus guards both ends: beforehand it trims the tool schema sent to the LLM; afterward it backstops with a synthesized error. Remember the order of these three blocks — rule declaration → solver computes live → violation backstop — the three sections below unfold exactly along it.

def get_allowed_tool_names(self, available_tools, error_on_empty=True,
                           last_function_response=None) -> list:
    # first step with init rules configured -> force only the init tools
    if not self.tool_call_history and self.init_tool_rules:
        return [r.tool_name for r in self.init_tool_rules]

    # else: intersect each child + parent rule's "legal set"
    relevant = self.child_based_tool_rules + self.parent_tool_rules
    if not relevant:
        return list(available_tools)
    allowed = set.intersection(*[
        r.get_valid_tools(self.tool_call_history, available_tools,
                           last_function_response)
        for r in relevant
    ])
    return list(allowed & set(available_tools))   # then & the available tools once more

Once the legal set is computed, V3's _get_valid_tools uses it to trim the batch of tool JSON schemas actually sent to the LLM — illegal tools never appear before the model's eyes. Meanwhile should_force_tool_call() decides whether this step is forced (compelling the model to call a tool, e.g. the init first step) or auto (letting the model roam free); terminal tools are handled through runtime_override_tool_json_schema (the rewrite gate seen in lesson 15).

🧠The legal set is dynamic: each time a tool executes, the solver records it into tool_call_history via register_tool_call, and the "legal set" that the next step's child / parent / rate-limit rules see narrows accordingly. So each step's "callable range" is computed and trimmed live, not fixed at the start — which is also why get_valid_tools must pass in both "the call history" and "the last return."

letta/agents/helpers.py_build_rule_violation_result: synthesize an error on a violation (simplified)
def _build_rule_violation_result(tool_name, valid_tools, tool_rules_solver):
    # note: the error string is built in the agent layer, not in the solver
    hint = tool_rules_solver.guess_rule_violation(tool_name)
    msg = (f"[ToolConstraintError] Cannot call {tool_name}, "
           f"valid tools include: {valid_tools}.")
    if hint:
        msg += f" Hint: {hint}"
    # don't execute the tool; feed this "error" back as its return to the model
    return ToolExecutionResult(status="error", func_return=msg)

Why is this move elegant? Because it turns a "violation" from a crash into a lesson. Better still, it dovetails with lesson 14: a violation still counts as "a tool was called this round" — so _decide_continuation's "called a tool → continue" criterion holds, the loop naturally takes another lap, and the chance to correct is handed right back to the model. That minimalist loop in lesson 14 dared to be minimal precisely because of this.

💡 Design highlight
Tool rules upgrade "how an agent should use tools" from the model's conscientiousness into a declarative, enforceable state machine — and two beautiful designs hide inside. First, the constraint is "computed": each step the ToolRulesSolver intersects the legal sets of all relevant rules to get "only these few are allowed right now," then trims the tool schema sent to the LLM accordingly — illegal tools never enter the model's view, cutting mis-selection at the source. Second, a violation is a lesson, not a crash: should the model still pick an illegal tool, the framework doesn't execute it but returns [ToolConstraintError] Cannot call X, valid tools include: [...] as that tool's "return value," letting the model read it and correct itself. The constraint thus guards both ends — shrink the schema beforehand, synthesize an error as a backstop afterward — disciplining an "unleashed LLM" into an "executor that works to spec." This is exactly why lesson 14's minimalist loop ("called a tool → continue") dared to be minimal: tool rules shouldered the grunt work for it.
🔬 Down to the code
Four anchors, one chain. The nine subclasses are in schemas/tool_rule.py (each pinning a Literal type — a discriminated union keyed on type), the enum roster in schemas/enums.py::ToolRuleType, and the base-class interface is BaseToolRule.get_valid_tools plus the property requires_force_tool_call. The solver is helpers/tool_rule_solver.py::ToolRulesSolver: model_post_init buckets, the core is get_allowed_tool_names, plus is_terminal_tool / has_children_tools / is_continue_tool / has_required_tools_been_called / is_requires_approval_tool / should_force_tool_call / register_tool_call / guess_rule_violation. The violation synthesis is in agents/helpers.py::_build_rule_violation_result. The wiring is in V3: letta_agent_v3.py::_get_valid_tools (trim the schema), _handle_ai_response (check tool_rule_violated), _decide_continuation (read terminal / child / continue / required); the solver hangs on the V2 base, self.tool_rules_solver = ToolRulesSolver(tool_rules=agent_state.tool_rules).
⚠️ Common pitfalls
Four points not to misremember. First, a violating tool is not executed — _build_rule_violation_result swaps it for a synthesized error fed back to the model, not "actually run, then errored." Second, get_allowed_tool_names intersects only child + parent rules to shrink the tool set; terminal / continue / required / approval don't shrink it — they act inside the loop. Third, the enum values are legacy-named: run_first / exit_loop / constrain_child_tools…, not Init / Terminal / Child. Fourth, Letta does no cycle / conflict detection — only per-rule pydantic validation (ChildToolRule.child_arg_nodes ⊆ children; ConditionalToolRule needs at least one mapping) plus agent_manager_helper.py::check_supports_structured_output erroring when more than one init rule is configured (for non-structured-output models). Whether rules clash, the framework won't cover for you — the configurer must think it through. A reminder, too: tool rules are a double-edged sword — wielded well they orchestrate precisely, misconfigured there's no fuse to stop you.

Digging a little deeper

The nine rules, each in one line with a mini example
  • run_first (Init): force the first step — archival_memory_search before send_message, look something up before you open your mouth.
  • exit_loop (Terminal): calling it wraps up — send_message is often set as the terminal tool.
  • continue_loop (Continue): calling it forces another lap, not letting the loop stop.
  • conditional: pick the next step by this tool's output — like a state machine's transition function, child_output_mapping maps an output value to the next tool.
  • constrain_child_tools (Child): after plan you may only call the execute children, welding "plan first, then execute" in place.
  • max_count_per_step: rate-limit — stop the model from spamming the same tool ten times in one step.
  • parent_last_tool (Parent): these children may only be called immediately after a given parent tool.
  • required_before_exit: must have been called before exit — e.g. log_result must run before finishing.
  • requires_approval: pause for a human to approve before calling — a safety catch for dangerous operations like dropping a database or transferring money.
How does the conditional rule pick the next step by "output value"?

Example: a classify tool returns "refund" or "complaint". ConditionalToolRule's child_output_mapping maps "refund" to process_refund and "complaint" to escalate, with default_child as the fallback for the unmatched case.

Why designed this way: it makes "who to call next" a function of this tool's output — effectively a built-in state machine at the tool-rule layer, so the model needn't remember the branching logic itself.

How it enters the legal set: it's bucketed into child_based_tool_rules, and get_valid_tools reads last_function_response (the previous tool's return) to decide which child to allow — exactly why the legal set must pass "the last return" in too.

Where in source: schemas/tool_rule.py::ConditionalToolRule (a pydantic validator requires at least one mapping); bucketing in tool_rule_solver.py::model_post_init.

Why does the "legal set" use intersection, not union?

Example: configure both a ChildToolRule (after A, only B/C) and a MaxCountPerStepToolRule (C at most once per step, and already called). Both are active: {B,C} ∩ ({B,C} minus the rate-limited C) = {B}.

Why designed this way: when several rules are active at once, "legal" must satisfy all of them simultaneously — intersection means "only what they all allow counts." A union would let through a tool that some rule plainly forbids, rendering the constraint toothless.

Where in source: in tool_rule_solver.py::get_allowed_tool_names, set.intersection(*[r.get_valid_tools(...) for r in child_based + parent]) then & available.

Why don't rules like terminal / required shrink the tool set?

Example: a TerminalToolRule marks send_message as terminal, but doesn't remove it from "callable tools" — you can of course still call it; it's just that after calling it the loop stops.

Why designed this way: shrinking the tool set (get_allowed_tool_names) answers one question only — "which tools are legally allowed next." Terminal / continue / required / approval govern "how the loop proceeds after a call" — actions inside the loop, not a tool filter. Keeping the two apart keeps the logic clean.

Where in source: only child_based_tool_rules + parent_tool_rules enter the intersection; the other buckets are read in _decide_continuation (lesson 14).

Why "synthesize an error" on a violation instead of raising an exception?

Example: the model ignores the constraint and picks an illegal tool. A bare raise would crash the whole step, and the user sees a baffling error.

Why designed this way: _build_rule_violation_result turns a violation into a status="error" tool return — its content is [ToolConstraintError] Cannot call X, valid tools include: [...]. Reading that "return," the model knows it picked wrong and which few to re-pick from. The loop isn't interrupted; the model self-corrects.

Connecting to lesson 14: a violation still counts as "a tool was called this round" — so _decide_continuation rules "called a tool → continue," the loop takes another lap, and the model gets its chance to correct. That's the backing behind lesson 14's minimalist loop daring to be minimal.

Where in source: agents/helpers.py::_build_rule_violation_result; the solver only supplies hints via guess_rule_violation.

The full requires_approval flow (continuing lesson 14's stop reasons)

Example: the model wants to call delete_all_records, a tool configured with requires_approval.

Flow: is_requires_approval_tool(name) is true → the loop doesn't execute the tool but stops, records the stop reason StopReasonType.requires_approval, and persists an "approval request" message built by create_approval_request_message_from_llm_response(...).

Resume: after a human approves, the next entry is an is_approval_response, which bypasses the violation check and lets execution through.

Connecting to lesson 14: this is one of the stop reasons lesson 14 listed — requires_approval lets the loop pause gracefully at "waiting for a human" rather than barging through a dangerous operation.
