自定义工具沙箱与安全修复The tool sandbox & a security fix

第 19 课结尾留了个悬念：一个自定义工具——用户自己写的 Python 源码——默认会被交给 SandboxToolExecutor，也就是"扔进沙箱"。可沙箱凭什么敢跑一段陌生人的代码？跑完之后，又该怎么把结果安全地收回服务端？

这是第五部分的收尾，也是整套工具系统"安全观"的落点。我们要把一张信任边界图画清楚：哪段路可以"放心传"、哪段路"必须验真"，以及一次真实的安全修复（PR #3343）如何把这条边界焊死。

🔌 生活类比

把沙箱想成监狱的探视窗。你（服务端）要递东西进去——那是你自己的物品，可以直接递原物。因为风险方向是"往里走"：东西进了高墙，再危险也跑不到你这边。

可对方（沙箱里跑的陌生代码）从里面递出来的任何包裹，你绝不能伸手直接接管。必须隔着玻璃、过安检、核对封条——因为你根本不知道里面是糖果还是刀片。

这一课全部的安全感，就压在这条朴素规矩上：往里递可以是原物，往外收必须过安检。剩下的工程，只是把"原物"翻译成 pickle、把"安检"翻译成 JSON 加一层校验而已。

这条规矩看着简单，却常被违反：很多"沙箱"只盯着把代码关进去，却忘了"收结果"这一步同样跨越边界。真正的安全，恰恰藏在你怎么把东西拿出来。

🌍 宏观理解

记住一句话就够了：自定义工具是不可信代码。它不在主进程里跑，而是被丢进一个沙箱——本地 venv、E2B 或 Modal，由 SandboxType 三选一。

沙箱内外只有两条数据通道，方向相反、信任也相反。server→sandbox 走 pickle（受信：服务端把自己造的 agent_state 序列化进去）；sandbox→server 走 JSON（不可信：绝不 pickle.loads），并加 marker+长度+MD5 帧来校验完整性。

这张信任边界图就是本课的全部。而 PR #3343 正是一次把回程通道从 pickle 改成 JSON 的真实安全修复——它把这条边界从"差不多安全"变成"焊死"。

再强调一遍方向感：数据从服务端流向沙箱时，权限是在收窄（进了低信任区），所以 pickle 安全；数据从沙箱流回服务端时，权限在放大（回到高信任区），所以必须只用 JSON。把"权限放大的方向"当成红线，整张图就记住了。

沙箱三选一：代码到底在哪跑

一个自定义工具被执行前，Letta 先决定"在哪跑"。这不是随机的，而是一条短路式的判定链：先看工具自己的偏好，再看全局配置，最后兜底到本地。

1

看 tool.metadata

若工具显式标了 sandbox=="modal" 且 Modal 已启用 → 用 Modal（云端容器，强隔离）。

2

看全局配置

否则查 ToolSettings.sandbox_type：若配了 e2b_api_key → 用 E2B（云端微沙箱）。

3

兜底本地

都没有 → Local：在本机建一个隔离 venv，用子进程跑。开发期最常见。

三种沙箱不是冗余，而是隔离强度与部署成本的权衡。本地 venv 零依赖、最快，但和服务端共享内核，隔离最弱；E2B、Modal 把代码送进云端独立容器，隔离强，但要联网和配额。所以同一段工具，开发期可能跑在本地，生产期被切到 Modal——这正是"在哪跑"必须可配置的原因。

💡类型定义在 schemas/enums.py::SandboxType（E2B / MODAL / LOCAL）；选择逻辑落在 sandbox_tool_executor.py，全局开关是 settings.py::ToolSettings.sandbox_type。关键点是：本课的"生成脚本 + 信任边界"主要讲本地沙箱这条路；marker+长度+MD5 帧是本地特有（E2B 改用 base64、Modal 走原生部署直接返回结构化 dict、连脚本都不生成）。但"信任边界"这条原则三种沙箱通用——回程数据一律不当可信代码反序列化。

顺便提一句配置：LocalSandboxConfig 还能调 sandbox_dir、use_venv、venv_name、pip_requirements 等，决定本地沙箱具体怎么落地。但这些都是"怎么跑"的旋钮，不改"信谁"的边界。

生成的沙箱脚本：把工具包起来跑

沙箱不会"直接调用"用户函数。服务端会现拼一段 Python 脚本，把所有需要的东西内联进去，再让沙箱整段执行。这段脚本由 tool_sandbox/base.py::_render_sandbox_code 拼出来。

这种"现拼脚本"的做法有个好处：服务端对沙箱里要发生的一切完全掌控——传什么、调用什么、怎么打包结果，全写死在脚本里，沙箱只是个老老实实的执行器。

1

解出 agent_state

pickle.loads(...) 还原服务端传进来的 agent_state（受信方向）。

2

内联参数

每个调用参数按 repr() 写成字面量，不是 pickle。

3

内联用户源码

把工具的 source_code 逐字粘进来，定义出函数本体。

4

调用 + 打包

_function_result = tool(...)，再把结果 + agent_state 打成 JSON。

5

加帧写 stdout

前面缀上 marker+长度+MD5，写进标准输出，等服务端来读。

letta/services/tool_sandbox/base.py_render_sandbox_code（简化骨架）
# 服务端把 agent_state pickle 进脚本，沙箱里 loads（受信方向）
agent_state = pickle.loads(b"...")
x = "<内联字面量>"            # 参数按 repr() 内联，不是 pickle
# <用户工具源码逐字内联>
_function_result = my_tool(x, agent_state=agent_state)
payload = _letta_json.dumps({"results": _function_result,
    "agent_state": agent_state.model_dump(mode="json")}).encode()  # JSON，不是 pickle
sys.stdout.buffer.write(MARKER + struct.pack(">I", len(payload)) + md5_hex + payload)
💡为什么把源码逐字内联、而不是 import 用户模块？因为沙箱里根本没有那个模块——工具源码只存在数据库里。内联后脚本自包含，沙箱只要一个干净解释器就能跑。脚本里还会用 coerce_dict_args_by_annotations(...) 按注解把参数强转成正确类型，避免字符串和数字错配。
⚠️agent_state 走 pickle.dumps（服务端造、沙箱 loads）；但调用参数是内联字面量（repr()），不是 pickle。整段脚本里，只有 agent_state 这一项用到了 pickle——这个区分后面会反复用到。

信任边界：方向决定序列化

现在把两条通道并排放。同一份脚本，两个方向用了两种序列化——这不是随意，而是信任在说话。

⬇️ server → sandbox

用 pickle 传 agent_state。受信：是服务端 pickle 自己造的对象，沙箱只负责 loads。权限往低信任侧流，没有"反序列化不可信输入"的问题。

⬆️ sandbox → server

只用 JSON + marker+长度+MD5。不可信：沙箱刚跑完任意用户代码，它的 stdout 绝不能 pickle.loads，只能当数据读。

为什么向下可以 pickle？因为被反序列化的对象是服务端自己造的，沙箱拿到的是数据、不是攻击面；就算沙箱想搞鬼，它也改不了"已经 pickle 进去的字节"。

为什么向上必须 JSON？因为 pickle.loads 会在反序列化时执行任意代码。沙箱里跑的是陌生人的工具，它的输出天然可疑——用 json.loads 读，最坏也只是拿到一坨假数据，而不是被 RCE。

⚠️一个反直觉点：受信方向用 pickle 不是"将就"，而是主动选择——它能一次性还原 agent_state 这种复杂对象，省去手写序列化。pickle 本身不是坏人，把它用在不可信方向才是。
⚠️举个具体攻击：恶意工具在 return 时不返回普通 dict，而是返回一个带 __reduce__ 的对象。若服务端用 pickle.loads 读它，反序列化的瞬间就执行攻击者指定的命令——服务端沦陷。改成 json.loads 后，这种对象根本无法表达，攻击面被结构性消除。
🔐⬇️ pickle→📦自己人，放行
🔐⬆️ JSON→🔍外人，过安检 MD5
信任＝方向：能 pickle.loads 你自己造的数据；绝不 pickle.loads 跨过不可信边界回来的数据。

回程帧：marker + 长度 + MD5

沙箱的 stdout 是一条嘈杂的河：用户代码可能 print 调试信息、可能抛异常栈、甚至可能故意打一个假结果。所以真正的结果不是"整段 stdout"，而是被一段帧圈起来的那一小块。

字段大小作用
MARKER16 字节uuid5 生成的定位锚，在噪声里标出"真结果从这里开始"
LENGTH4 字节 '>I'big-endian 无符号整数，告诉你 payload 有多长，好精确切片
MD532 字节 hexpayload 的校验和，抓篡改与截断
JSON_PAYLOADLENGTH 字节真正的结果：工具返回值 + agent_state 的 JSON

MARKER(16)·LENGTH(4)·MD5(32)·JSON_PAYLOAD

pos = data.find(MARKER)                       # 在嘈杂 stdout 里定位真结果
length = struct.unpack(">I", data[p:p+4])[0]
payload = data[start : start+length]
if hashlib.md5(payload).hexdigest() != checksum:
    raise Exception("Function ran, but output is corrupted.")
# —— 下面两行其实在 helpers/tool_parser_helper.py::parse_stdout_best_effort 里 ——
result = json.loads(payload)                     # JSON，绝不 pickle.loads
agent_state = AgentState.model_validate(result["agent_state"])  # pydantic 重水合

你可能会问：为什么用 stdout 这种"脏"通道，而不开一条干净的返回管道？因为沙箱的边界本质是一个进程，进程之间最通用、最不挑运行时的桥就是标准输出。代价是 stdout 里混着一切，于是必须靠帧把"真结果"从噪声里捞出来。

💡这套帧防的是什么？用户工具能往 stdout 打任何东西，包括一个伪造的结果或假 marker。marker 圈出真 payload，长度做精确切片，MD5 抓篡改/截断——三层叠起来，服务端才敢说"我读到的就是工具真正返回的那一份"。

顺序也有讲究：先比 MD5、再 json.loads。若先解析、出错了再说，攻击者就有机会用畸形 JSON 去触发解析器的边角行为；先校验完整性，等于把"可疑输入"挡在解析之前。

PR #3343：把 pickle 换成 JSON

帧校验一直都在，但有一处老漏洞：早期回程 payload 是 pickle 编码的，服务端读回来时会 pickle.loads。这等于对"刚跑完任意用户代码的沙箱输出"做反序列化——一个干净利落的服务端 RCE。

letta/services/helpers/tool_parser_helper.pyPR #3343 · commit 1131535
# 旧（有漏洞）：服务端对沙箱 stdout 直接 pickle.loads -> 服务端 RCE
- result = pickle.loads(text)
# 新（已修复）：只 json.loads，绝不执行代码
+ result = json.loads(payload)
⚠️注意分寸：marker+长度+MD5 帧早于 #3343 就存在。#3343 只把 sandbox→server 的 payload 编码从 pickle 换成 JSON；server→sandbox 仍然 pickle agent_state——那是有意保留的受信方向，不是漏改。

这条修复的价值不在代码量，而在思路：序列化格式的选择本身就是一道安全闸。把 pickle 留给"我造的、往低信任侧流"的数据，把 JSON 留给"别人造的、往高信任侧回"的数据——这条规矩一旦内化，类似的 RCE 在设计阶段就被挡住，而不必等事后修补。

把回程在脑子里走一遍

抽象讲完，拿个具体例子收一收。假设用户写了个 get_weather(city) 工具，模型在某一步决定调用它、参数是 city="上海"。从这一刻起，发生的事是这样的。

服务端先把当前 agent_state 用 pickle.dumps 成一段字节，连同 city 的字面量、用户那段 get_weather 源码，一起拼进一份临时脚本。

💡这一步就能看清区分：agent_state 是 pickle 字节，而 city 只是脚本里一个内联的字符串字面量 "上海"。同一份脚本，两种"塞数据进去"的方式，对应两种信任。

脚本被丢进沙箱执行：沙箱里 pickle.loads 还原 agent_state、跑 get_weather("上海")、拿到返回值，再把"返回值 + 更新后的 agent_state"打成 JSON，前面缀上 marker、长度、MD5，整包写进 stdout。

🧷这里是全课的转折点：从下一步起，所有数据都跨过了不可信边界。沙箱刚执行完一段我们无法预先信任的代码，它写出来的每个字节，都得当"外人递出的包裹"来对待。

服务端读回 stdout：先用 marker 定位、按长度切片、比对 MD5——任何一步不符，就判定结果损坏、直接抛错。校验通过后，只对 payload 做 json.loads。

最后用 AgentState.model_validate 把状态重新水合成对象。全程没有任何一行对沙箱输出做 pickle.loads——这就是整条旅程的安全保证：往里是 pickle、往外是 JSON，边界两侧泾渭分明。

🧷把这趟旅程压缩成一句话：数据下行用 pickle、上行用 JSON，中间隔着一道 marker+长度+MD5 的安检门。记住这句，等于记住了整课。
💡 设计亮点

一句话收束：你信任哪个方向，决定你用哪种序列化。

pickle.loads 能在反序列化时执行任意代码，于是规则被砍到极简：可以 pickle.loads 你自己造的数据（server→sandbox，权限往下流进低信任沙箱，没风险）；绝不 pickle.loads 跨过不可信边界回来的数据（sandbox→server，刚跑完任意用户代码，只能当 JSON 读）。

PR #3343 修的正是这个：旧代码 pickle.loads 沙箱 stdout——服务端 RCE；改成 json.loads 就堵死了。而 marker+长度+MD5 是另一层——完整性层：用户工具能打假结果，marker 圈真 payload、MD5 抓篡改。

安全被化简成一个词：信任＝方向。这也收口了整条工具链：第 18 课"不跑代码就读它" → 第 19 课"自定义→沙箱" → 第 20 课"连沙箱的输出也不可信"。

顺着这条线还能预判别处：只要看到"反序列化不可信来源"，就该立刻警觉；只要确认"数据是自己造、往低信任侧送"，就可以放心用高表达力的格式。这套直觉，远不止用在沙箱上，它几乎是所有跨边界通信的底色。

🔬 落到代码

生成脚本：tool_sandbox/base.py::AsyncToolSandboxBase.generate_execution_script 与 _render_sandbox_code。本地执行：local_sandbox.py::AsyncToolSandboxLocal（建/复用 venv、子进程、超时）。

回读校验：local_sandbox.py::parse_out_function_results_markers 找 marker、切长度、比 MD5；再交 helpers/tool_parser_helper.py::parse_stdout_best_effort 做 json.loads + AgentState.model_validate。

类型与安全：沙箱类型在 schemas/enums.py::SandboxType；本次安全修复是 PR #3343（commit 1131535）。

顺带一提，local_sandbox.py 里还藏着 venv 复用、依赖安装、超时控制等工程细节；它们不影响信任边界这条主线，却决定了本地沙箱跑得快不快、稳不稳。

⚠️ 常见误区

误区一：以为脚本由 .j2 模板渲染。 letta/templates/sandbox_code_file.py.j2 在 v0.16.8 未被任何 .py 引用，只能当对照参考；真实脚本由 _render_sandbox_code 拼字符串。

误区二：以为参数也走 pickle。 参数是内联字面量（repr()），整段脚本里只有 agent_state 一项走 pickle。

误区三：以为帧是 #3343 加的。 marker+长度+MD5 帧早就存在，#3343 只改了 payload 编码（pickle→JSON）。

误区四：信 docstring 和变量名。 generate_execution_script 的 docstring 写着"base64-encode/pickle the result"、变量名带 _pkl——都是遗留，实际装的是 JSON。

再挖深一点

下面四个抽屉，分别回答"为什么这样切边界""帧到底防谁""#3343 改了哪一段""本地 venv 怎么跑"。想细抠的同学可以逐个展开。

① 信任边界为什么这样切？

核心只有一条：pickle.loads 会在反序列化时执行任意代码。所以"能不能 pickle.loads"等价于"我信不信这段字节的来源"。

server→sandbox 的 agent_state 是服务端自己 pickle.dumps 的，来源可信，沙箱只 loads——风险往低信任侧流，没问题。sandbox→server 的输出来自陌生代码，来源不可信，所以只能 json.loads。

再换个角度：信任不是"信不信这段数据长得对"，而是"信不信造它的人"。pickle 危险，恰恰因为它把"数据"和"可执行代码"混在一起——来源可信时这种混合很方便（能直接还原复杂对象），来源不可信时它就成了炸弹。

② marker+长度+MD5 防的是什么？

防的是"结果通道被污染"。沙箱的 stdout 谁都能写：调试 print、异常栈、甚至一个伪造的 marker 加假结果。

于是三层各司其职：marker 在噪声里定位真 payload 的起点；length 精确切出 payload，避免被尾部噪声带偏；MD5 比对校验和，一旦被截断或篡改就 raise Exception("Function ran, but output is corrupted.")。

还有个细节：marker 是 uuid5 生成的 16 字节，几乎不可能被用户输出意外撞上，也很难被恶意"猜中"，于是"定位"这一步本身就带了一点防伪意味。

有人会问：MD5 不是不安全吗？这里它只用来抓意外损坏，并不抗密码学攻击——真正挡恶意的是"只 json.loads、绝不 pickle.loads"那条规矩，MD5 只是完整性的廉价快照。

③ PR #3343 到底改了什么？

只改了一件事：把 sandbox→server 的 payload 编码从 pickle 换成 JSON，于是服务端读回来时从 pickle.loads 变成 json.loads，RCE 面被消除。

没改的：marker+长度+MD5 帧（本就存在）、server→sandbox 仍 pickle agent_state（有意的受信方向）。另外 safe_pickle.py::safe_pickle_dumps 只服务于 modal_sandbox_v2，加的是 10MB/深度 50 的防崩溃护栏，并非防恶意。

一句话区分两件事：帧负责完整性（结果没被改坏），编码格式负责安全性（读它不会执行代码）。#3343 动的是后者，前者从一开始就在。

想直观点：修复前服务端这端是 pickle.loads(沙箱字节)，修复后是 json.loads(沙箱字节)。同样在读不可信输入，前者能被反序列化触发 RCE，后者最多解析失败——差别就这么大。

④ 本地 venv 怎么跑？E2B/Modal 有何不同？

本地 AsyncToolSandboxLocal：按需 venv.create(with_pip=True) 建或复用虚拟环境、pip install -r 装依赖，脚本写临时 .py，用 asyncio.create_subprocess_exec 起子进程跑，超时是 tool_sandbox_timeout（默认 180s）。

差异不止在外层：本地用 marker+长度+MD5 帧；E2B 改用 base64 编码（不是多包一层，是另一条路径）；Modal 直接返回结构化 dict，连生成脚本和帧都没有。三者真正共用的是"信任边界"原则——去程可信、回程一律 JSON / 结构化、绝不 pickle.loads。

也正因为本地沙箱与服务端共享内核，它的隔离是"够用级"而非"强保证"：能挡住误伤和大多数意外，但要跑完全不可信的代码，应切到 E2B 或 Modal 的容器级隔离。

超时到了会怎样？子进程被杀、本次调用按失败处理，不会把服务端拖死。临时脚本文件用完即清，venv 则可复用，省去每次重新建环境的开销。

✅ 本课要点
  • 自定义工具是不可信代码，跑在沙箱里（local / E2B / Modal，由 SandboxType 三选一）。
  • server→sandbox = pickle（受信，传 agent_state）；sandbox→server = JSON（不可信）+ marker+长度+MD5 帧。
  • 绝不 pickle.loads 沙箱输出——这正是 PR #3343（commit 1131535）的修复。
  • .j2 模板在 v0.16.8 未被引用，真实脚本在 _render_sandbox_code；docstring 与 _pkl 命名是遗留。
  • 一个词记住全部：信任＝方向。

Lesson 19 ended on a cliffhanger: a custom tool — Python source the user wrote themselves — is by default handed to SandboxToolExecutor, i.e. "thrown into a sandbox." But what gives the sandbox the nerve to run a stranger's code? And once it finishes, how do you safely collect the result back into the server?

This is Part 5's finale, and the landing point for the whole tool system's "security mindset." We will draw one trust-boundary diagram clearly: which leg of the trip you can "pass along freely," which leg "must be verified," and how one real security fix (PR #3343) welds that boundary shut.

🔌 Real-world analogy

Think of the sandbox as a prison visitation window. You (the server) want to pass something in — it is your own belongings, so you can hand over the original object. Because the risk only flows inward: once something is behind the high wall, however dangerous, it can never reach your side.

But any parcel the other party (the stranger's code running inside) hands out, you must never reach in and take directly. It has to go through the glass, through the scanner, with its seal checked — because you have no idea whether it holds candy or a razor blade.

All of this lesson's sense of safety rests on one plain rule: passing in may be the original; collecting out must pass the scanner. The rest of the engineering merely translates "the original" into pickle and "the scanner" into JSON plus a layer of verification.

The rule looks simple yet is often broken: many "sandboxes" fixate on locking code in, yet forget that "collecting the result" crosses the boundary just the same. Real safety hides precisely in how you take things back out.

🌍 The big picture

One sentence is enough: a custom tool is untrusted code. It does not run in the main process; it is dropped into a sandbox — a local venv, E2B, or Modal, one of three picked by SandboxType.

Across the wall there are only two data channels, opposite in direction and opposite in trust. server→sandbox uses pickle (trusted: the server serializes the agent_state it built itself); sandbox→server uses JSON (untrusted: never pickle.loads), with a marker+length+MD5 frame to verify integrity.

This trust-boundary diagram is the whole lesson. And PR #3343 is exactly the real security fix that switched the return channel from pickle to JSON — taking this boundary from "roughly safe" to "welded shut."

Stressing the sense of direction once more: when data flows from server to sandbox, privilege narrows (entering a low-trust zone), so pickle is safe; when data flows from sandbox back to server, privilege widens (returning to a high-trust zone), so it must be JSON only. Treat "the direction in which privilege widens" as the red line, and you have memorized the whole diagram.

Three sandboxes, one pick: where does the code actually run?

Before a custom tool runs, Letta first decides "where to run it." This is not random but a short-circuit decision chain: check the tool's own preference first, then the global config, and finally fall back to local.

1

Check tool.metadata

If the tool explicitly marks sandbox=="modal" and Modal is enabled → use Modal (cloud container, strong isolation).

2

Check the global config

Otherwise consult ToolSettings.sandbox_type: if an e2b_api_key is set → use E2B (cloud micro-sandbox).

3

Fall back to local

Neither → Local: build an isolated venv on the host and run it in a subprocess. The most common case during development.

The three sandboxes are not redundant; they are a trade-off between isolation strength and deployment cost. A local venv has zero dependencies and is fastest, but shares the kernel with the server and isolates the least; E2B and Modal ship code into a dedicated cloud container — strong isolation, yet they need network access and quota. So the same tool may run locally in development and be switched to Modal in production — exactly why "where to run" must be configurable.

💡The type is defined in schemas/enums.py::SandboxType (E2B / MODAL / LOCAL); the selection logic lives in sandbox_tool_executor.py, and the global switch is settings.py::ToolSettings.sandbox_type. The key point: this lesson's "generate script + trust boundary" walkthrough is mainly the local path; the marker+length+MD5 frame is local-only (E2B uses base64 instead, and Modal returns a native structured dict via deployment — no generated script at all). But the trust-boundary principle is universal across all three — none of them deserializes the return as trusted code.

A configuration aside: LocalSandboxConfig can also tune sandbox_dir, use_venv, venv_name, pip_requirements and more, deciding how the local sandbox is actually laid down. But these are all "how to run" knobs; they do not change the "whom to trust" boundary.

The generated sandbox script: wrap the tool and run it

The sandbox does not "directly call" the user's function. The server assembles a Python script on the fly, inlines everything it needs, and then lets the sandbox execute the whole thing. That script is assembled by tool_sandbox/base.py::_render_sandbox_code.

This "assemble-the-script" approach has an upside: the server has full control over everything that happens inside the sandbox — what gets passed in, what gets called, how the result is packed are all hard-coded into the script, and the sandbox is merely an obedient executor.

1

Unpickle agent_state

pickle.loads(...) restores the agent_state the server passed in (the trusted direction).

2

Inline the arguments

Each call argument is written as a literal via repr(), not pickle.

3

Inline the user source

The tool's source_code is pasted in verbatim, defining the function body.

4

Call + pack

_function_result = tool(...), then pack the result + agent_state into JSON.

5

Frame + write stdout

Prefix it with marker+length+MD5, write to standard output, and wait for the server to read it.

letta/services/tool_sandbox/base.py_render_sandbox_code (simplified skeleton)
# server pickles agent_state into the script; the sandbox loads it (trusted direction)
agent_state = pickle.loads(b"...")
x = "<inlined literal>"            # args inlined via repr(), not pickle
# <user tool source inlined verbatim>
_function_result = my_tool(x, agent_state=agent_state)
payload = _letta_json.dumps({"results": _function_result,
    "agent_state": agent_state.model_dump(mode="json")}).encode()  # JSON, not pickle
sys.stdout.buffer.write(MARKER + struct.pack(">I", len(payload)) + md5_hex + payload)
💡Why inline the source verbatim instead of importing the user's module? Because that module simply does not exist inside the sandbox — the tool source lives only in the database. Inlined, the script is self-contained, and the sandbox needs only a clean interpreter to run it. The script also calls coerce_dict_args_by_annotations(...) to coerce each argument to the right type per its annotation, avoiding string/number mismatches.
⚠️agent_state goes through pickle.dumps (server-built, sandbox loads); but the call arguments are inlined literals (repr()), not pickle. In the entire script, agent_state is the only thing that uses pickle — a distinction we will lean on repeatedly later.

Trust boundary: direction decides serialization

Now place the two channels side by side. One script, two directions, two serializations — not arbitrary, but trust speaking.

⬇️ server → sandbox

Uses pickle to pass agent_state. Trusted: the server pickles an object it built itself, and the sandbox only loads it. Privilege flows toward the low-trust side, so there is no "deserializing untrusted input" problem.

⬆️ sandbox → server

Uses only JSON + marker+length+MD5. Untrusted: the sandbox just ran arbitrary user code, so its stdout must never be pickle.loads'd — it can only be read as data.

Why can the downward leg use pickle? Because the object being deserialized was built by the server itself; the sandbox receives data, not an attack surface. Even if the sandbox wanted to misbehave, it cannot alter "the bytes already pickled in."

Why must the upward leg be JSON? Because pickle.loads executes arbitrary code during deserialization. The sandbox ran a stranger's tool, so its output is inherently suspect — read with json.loads, the worst case is a blob of fake data, not an RCE.

⚠️A counterintuitive point: using pickle on the trusted direction is not "making do" but a deliberate choice — it restores a complex object like agent_state in one shot, sparing hand-written serialization. pickle itself is not the villain; using it on the untrusted direction is.
⚠️A concrete attack: a malicious tool returns, instead of a plain dict, an object carrying a __reduce__. If the server reads it with pickle.loads, the moment of deserialization runs the attacker's chosen command — the server falls. Switch to json.loads and such an object cannot even be expressed; the attack surface is structurally eliminated.
🔐⬇️ pickle→📦Insider — let it through
🔐⬆️ JSON→🔍Outsider — scan with MD5
Trust = direction: you may pickle.loads data you built yourself; never pickle.loads data coming back across the untrusted boundary.

The return frame: marker + length + MD5

The sandbox's stdout is a noisy river: user code may print debug info, may throw a stack trace, may even deliberately print a fake result. So the real result is not "the whole stdout" but the small piece fenced off by a frame.

You might ask: why use a "dirty" channel like stdout instead of opening a clean return pipe? Because the sandbox boundary is essentially a process, and the most universal, least runtime-fussy bridge between processes is standard output. The cost is that stdout mixes in everything, so a frame is required to fish "the real result" out of the noise.

💡What does this frame defend against? A user tool can print anything to stdout, including a forged result or a fake marker. The marker fences off the real payload, the length slices it precisely, and MD5 catches tampering/truncation — stacked together, only then dare the server say "what I read is exactly the one the tool actually returned."

The order matters too: compare MD5 first, then json.loads. If you parsed first and handled errors later, an attacker could use malformed JSON to trigger the parser's corner-case behavior; verifying integrity first holds "suspicious input" back before parsing.

PR #3343: swap pickle for JSON

The frame check was always there, but one old hole remained: early on, the return payload was pickle-encoded, and the server would pickle.loads it on read-back. That amounts to deserializing "the output of a sandbox that just ran arbitrary user code" — a clean-cut server-side RCE.

letta/services/helpers/tool_parser_helper.pyPR #3343 · commit 1131535
# old (vulnerable): server pickle.loads on the sandbox stdout -> server RCE
- result = pickle.loads(text)
# new (fixed): json.loads only, never executes code
+ result = json.loads(payload)
⚠️Mind the scope: the marker+length+MD5 frame existed before #3343. #3343 only switched the sandbox→server payload encoding from pickle to JSON; server→sandbox still pickles agent_state — that is the trusted direction kept on purpose, not an oversight.

The fix's value is not in lines of code but in the mindset: the choice of serialization format is itself a security gate. Reserve pickle for "data I built, flowing to the low-trust side," and JSON for "data others built, returning to the high-trust side" — once internalized, this rule blocks similar RCEs at design time, instead of waiting to patch after the fact.

Walk the return trip in your head

Abstractions done, let us close with a concrete example. Suppose the user wrote a get_weather(city) tool, and at some step the model decides to call it with city="Shanghai". From this moment on, here is what happens.

The server first pickle.dumps the current agent_state into a blob of bytes and, together with the city literal and the user's get_weather source, assembles them into a temporary script.

💡This step already shows the distinction: agent_state is pickle bytes, while city is just an inlined string literal "Shanghai" in the script. One script, two ways of "stuffing data in," matching two kinds of trust.

The script is dropped into the sandbox to run: there it pickle.loads the agent_state, runs get_weather("Shanghai"), gets the return value, then packs "the return value + the updated agent_state" into JSON, prefixes marker, length, MD5, and writes the whole bundle to stdout.

🧷This is the lesson's turning point: from the next step on, all data has crossed the untrusted boundary. The sandbox just finished running code we could not trust in advance, so every byte it writes out must be treated as "a parcel handed over by an outsider."

The server reads stdout back: first locate by marker, slice by length, compare MD5 — if any step mismatches, the result is judged corrupted and an error is thrown outright. Once verification passes, only the payload is json.loads'd.

Finally AgentState.model_validate re-hydrates the state into an object. Nowhere along the way is a single line of pickle.loads applied to the sandbox output — that is the whole trip's safety guarantee: pickle going in, JSON coming out, the two sides of the boundary kept cleanly apart.

🧷Compress the trip into one sentence: data downstream uses pickle, upstream uses JSON, with a marker+length+MD5 security gate in between. Remember that one line and you have remembered the whole lesson.
💡 Design highlight

Close it in one sentence: which direction you trust decides which serialization you use.

pickle.loads can run arbitrary code during deserialization, so the rule is cut to the bone: you may pickle.loads data you built yourself (server→sandbox, privilege flowing down into the low-trust sandbox, no risk); never pickle.loads data coming back across the untrusted boundary (sandbox→server, just ran arbitrary user code, can only be read as JSON).

PR #3343 fixed exactly this: the old code did pickle.loads on the sandbox stdout — a server RCE; switching to json.loads seals it. The marker+length+MD5 is a separate layer — the integrity layer: a user tool can print a fake result, so the marker fences the real payload and MD5 catches tampering.

Security boils down to one word: trust = direction. This also closes the whole tool chain: Lesson 18 "read it without running it" → Lesson 19 "custom → sandbox" → Lesson 20 "even the sandbox's output is untrusted."

Following this line you can anticipate elsewhere too: whenever you see "deserializing an untrusted source," be instantly wary; whenever you confirm "the data is self-built, sent to the low-trust side," you may freely use a high-expressiveness format. This intuition reaches far beyond the sandbox — it is almost the baseline of all cross-boundary communication.

🔬 Down to the code

Script generation: tool_sandbox/base.py::AsyncToolSandboxBase.generate_execution_script and _render_sandbox_code. Local execution: local_sandbox.py::AsyncToolSandboxLocal (build/reuse venv, subprocess, timeout).

Read-back verification: local_sandbox.py::parse_out_function_results_markers finds the marker, slices the length, compares MD5; then hands off to helpers/tool_parser_helper.py::parse_stdout_best_effort for json.loads + AgentState.model_validate.

Types and security: the sandbox type is in schemas/enums.py::SandboxType; this security fix is PR #3343 (commit 1131535).

By the way, local_sandbox.py also hides engineering details like venv reuse, dependency install, and timeout control; they do not affect the trust-boundary throughline, yet they decide whether the local sandbox runs fast and stays stable.

⚠️ Common misconceptions

Misconception 1: thinking the script is rendered from the .j2 template. letta/templates/sandbox_code_file.py.j2 is not referenced by any .py in v0.16.8 — it only serves as a side reference; the real script is string-assembled by _render_sandbox_code.

Misconception 2: thinking arguments also go through pickle. Arguments are inlined literals (repr()); in the whole script only agent_state uses pickle.

Misconception 3: thinking the frame was added by #3343. The marker+length+MD5 frame already existed; #3343 only changed the payload encoding (pickle→JSON).

Misconception 4: trusting the docstring and variable names. The docstring of generate_execution_script says "base64-encode/pickle the result," and a variable carries _pkl — both are legacy; what is actually carried is JSON.

Dig a little deeper

The four drawers below answer, respectively, "why cut the boundary this way," "whom the frame really defends against," "which lines #3343 changed," and "how the local venv runs." If you want the fine detail, expand them one by one.

① Why is the trust boundary cut this way?

There is just one core idea: pickle.loads runs arbitrary code during deserialization. So "may I pickle.loads this" is equivalent to "do I trust the source of these bytes."

The server→sandbox agent_state is pickle.dumps'd by the server itself; the source is trusted, and the sandbox only loads it — risk flows to the low-trust side, no problem. The sandbox→server output comes from unfamiliar code; the source is untrusted, so it can only be json.loads'd.

Another angle: trust is not "do I believe this data looks right" but "do I trust whoever made it." pickle is dangerous precisely because it mixes "data" and "executable code" — when the source is trusted this mix is convenient (it restores complex objects directly), and when the source is untrusted it becomes a bomb.

② What does marker+length+MD5 defend against?

It defends against "the result channel being polluted." Anyone can write to the sandbox's stdout: debug prints, exception stacks, even a forged marker plus a fake result.

So the three layers each do their job: marker locates the start of the real payload amid the noise; length slices the payload precisely, avoiding being misled by trailing noise; MD5 compares the checksum, and on truncation or tampering it does raise Exception("Function ran, but output is corrupted.").

One more detail: the marker is a uuid5-generated 16 bytes, almost impossible for user output to hit by accident and hard to "guess" maliciously, so the "locate" step itself carries a touch of anti-forgery.

Someone may ask: isn't MD5 insecure? Here it only catches accidental corruption and does not resist cryptographic attacks — what truly blocks malice is the rule "json.loads only, never pickle.loads"; MD5 is merely a cheap integrity snapshot.

③ What exactly did PR #3343 change?

It changed just one thing: switching the sandbox→server payload encoding from pickle to JSON, so the server's read-back goes from pickle.loads to json.loads, and the RCE surface is eliminated.

What it did not change: the marker+length+MD5 frame (already there), and server→sandbox still pickling agent_state (the intentional trusted direction). Also safe_pickle.py::safe_pickle_dumps serves only modal_sandbox_v2, adding a 10MB/depth-50 anti-crash guardrail, not an anti-malice one.

One sentence to separate two things: the frame handles integrity (the result was not corrupted), and the encoding format handles security (reading it does not execute code). #3343 touched the latter; the former was there from the start.

For intuition: before the fix the server side was pickle.loads(sandbox bytes), after it is json.loads(sandbox bytes). Both read untrusted input, but the former can be deserialization-triggered into an RCE while the latter at most fails to parse — that is how big the difference is.

④ How does the local venv run? How do E2B/Modal differ?

Local AsyncToolSandboxLocal: on demand venv.create(with_pip=True) builds or reuses a virtual environment, pip install -r installs dependencies, the script is written to a temporary .py, and asyncio.create_subprocess_exec spawns a subprocess to run it, with the timeout being tool_sandbox_timeout (default 180s).

The differences are more than skin-deep: local uses the marker+length+MD5 frame; E2B uses base64 encoding instead (a different path, not an extra layer); Modal returns a structured dict directly, with no generated script or frame at all. What the three genuinely share is the trust-boundary principle — trusted on the way in, JSON/structured on the way back, never pickle.loads.

Precisely because the local sandbox shares the kernel with the server, its isolation is "good enough" rather than a "strong guarantee": it can block accidents and most mishaps, but to run fully untrusted code you should switch to the container-level isolation of E2B or Modal.

What happens when the timeout hits? The subprocess is killed and this call is treated as a failure, without dragging the server down. The temporary script file is cleaned up after use, while the venv can be reused, saving the cost of rebuilding the environment each time.

✅ Key points
  • A custom tool is untrusted code, run in a sandbox (local / E2B / Modal, one of three by SandboxType).
  • server→sandbox = pickle (trusted, passing agent_state); sandbox→server = JSON (untrusted) + a marker+length+MD5 frame.
  • Never pickle.loads the sandbox output — that is exactly the fix in PR #3343 (commit 1131535).
  • The .j2 template is unreferenced in v0.16.8; the real script lives in _render_sandbox_code; the docstring and the _pkl naming are legacy.
  • One word to remember it all: trust = direction.
